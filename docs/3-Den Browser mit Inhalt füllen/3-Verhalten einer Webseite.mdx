Nachdem wir in den vorherigen Kapiteln gelernt haben, wie wir mit HTML die Struktur und mit CSS das Aussehen einer Webseite definieren, fehlt noch ein entscheidendes Puzzleteil: **das Verhalten**.

Wie reagiert eine Webseite auf Benutzereingaben? Wie können wir Inhalte dynamisch laden, ohne die gesamte Seite neu zu laden? Wie speichern wir Daten im Browser? All diese Fragen beantworten wir mit **JavaScript** - der Programmiersprache des Webs.

In diesem Kapitel lernst du:
* Die Grundlagen von JavaScript kennen
* Wie du auf Benutzerinteraktionen reagierst (Events)
* Wie du den Inhalt einer Webseite dynamisch änderst (DOM Manipulation)
* Wie du Formulareingaben validierst
* Wie du Daten im Browser speicherst
* Wie du asynchron mit einem Server kommunizierst

JavaScript macht aus statischen HTML-Seiten **interaktive Webanwendungen**. Ohne JavaScript wäre das moderne Web, wie wir es kennen, undenkbar - kein Gmail, kein Google Maps, kein Facebook, kein Netflix. JavaScript ist die Sprache, die das Web zum Leben erweckt.

## 1. Javascript Basics

* **Primäre Sprache**: JavaScript ist die primäre Sprache zur Definition des browserseitigen Verhaltens von Webseiten.
* **Nichts mit Java zu tun**: JavaScript hat nichts mit Java zu tun, obwohl die Namen es vermuten lassen.
    * **Marketing-Trick**: Tatsächlich hat die Firma NetScape den Namen JavaScript gewählt, um von der Popularität von Java zu profitieren.
    * **Interpretiert vs. kompiliert**: Javascript ist eine interpretierte Sprache, die hauptsächlich im Browser ausgeführt wird, während Java eine kompilierte Sprache ist (bzw. streng genommen einen hybriden Ansatz verwendet), die auf verschiedenen Plattformen laufen kann.
    * **Dynamisch vs. statisch typisiert**: Javascript ist dynamisch typisiert, während Java statisch typisiert ist.
    * **Prototypenbasiert vs. objektorientiert**: Java ist objektorientiert, während Javascript prototypenbasiert ist.
* **Direkte Ausführung**: Browser können JavaScript-Code direkt ausführen

### Type Coercion

:::warning
Ab diesem Punkt kann und werde ich es mir nicht verkneifen, euch regelmäßig mit Javascript Eigenheiten zu geißeln.
:::

* Ein leider sehr wichtiges Thema in JavaScript ist die automatische Typumwandlung (Type Coercion).
* JavaScript versucht oft, Werte automatisch in einen anderen Typ umzuwandeln, wenn dies erforderlich ist.
* Das sieht man bspw. bei der Verwendung des Gleichheitsoperators `==`, der versucht, die Werte zu konvertieren, bevor er sie vergleicht.

<LiveJavascript
    initialCode={`
console.log(5 == '5'); // true, da '5' in eine Zahl umgewandelt wird
console.log(5 === '5'); // false, da die Typen unterschiedlich sind

console.log(0 == false); // true, da false in 0 umgewandelt wird
console.log(0 === false); // false, da die Typen unterschiedlich sind

console.log(null == undefined); // true, da beide als "kein Wert" betrachtet werden
console.log(null === undefined) // false, da die Typen unterschiedlich sind; undefined ist ein eigener Typ, null ist ein Objekt
`} />

* Es ist daher ratsam, den strikten Gleichheitsoperator `===` zu verwenden, um unerwartete Ergebnisse zu vermeiden.
* Das gleiche Verhalten ist bei anderen Operatoren zu beobachten, z.B. bei `+`, der sowohl für die Addition als auch für die String-Konkatenation verwendet wird.

<LiveJavascript
    initialCode={`
console.log(5 + '5'); // '55', da 5 in einen String umgewandelt wird
console.log(5 + 5); // 10, normale Addition
console.log('5' + '5'); // '55', String-Konkatenation
`} />

:::info
**Warum kommt beim vorherigen Beispiel "55" und nicht 10 heraus?**

Es wurde in der Standard-JavaScript-Spezifikation (ECMAScript, verwaltet durch die Ecma International) einfach definiert, dass wenn mindestens einer der Operanden ein String ist, eine String-Konkatenation durchgeführt wird.
Ist halt so.
Daher wird die Zahl 5 in einen String umgewandelt und die beiden Strings werden zusammengefügt.
:::

* Das macht das Arbeiten mit JavaScript manchmal unvorhersehbar und schwierig zu warten/debuggen.

### Funktionen

* Function als Deklaration

```javascript
function add(a, b) {
  return a + b;
}
```

* Function als Ausdruck

```javascript
const add = function(a, b) {
  return a + b;

```

* Arrow Function (kürzere Syntax, kein eigenes `this`)

```javascript
const add = (a, b) => a + b;

```

<LiveJavascript
    initialCode={`
function add(a, b) {
  return a + b;
}
const multiply = function(a, b) {
  return a * b;
};
const subtract = (a, b) => a - b;
console.log(add(5, 3)); // 8
console.log(multiply(5, 3)); // 15
console.log(subtract(5, 3)); // 2
`} />

:::info
**Warum kein eigenes `this`?**

In JavaScript bezieht sich `this` innerhalb einer Funktion auf das Objekt, von dem die Funktion aufgerufen wird.
Innerhalb einer Arrow Function hingegen wird `this` lexikalisch gebunden.
Das heißt: Der Wert von `this` in einer Arrow Function wird *nicht* durch den Aufrufkontext bestimmt, sondern durch den umgebenden Kontext/Scope, in dem die Arrow Function definiert wurde.

Beispiel:

<LiveJavascript
    initialCode={`
this.value = 10;

const obj = {
  value: 42,
  regularFunction: function() {
    console.log(this.value);
    // 'this' wird bei regulären Funktionen dynamisch durch den Aufrufkontext bestimmt.
    // Das heißt, wenn die Funktion als Methode des Objekts aufgerufen wird ("obj.regularFunction()"), bezieht sich 'this' auf 'obj'.
    // Ausgabe: 42
  },
  arrowFunction: () => {
    console.log(this.value);
    // 'this' wird bei Arrow Functions lexikalisch gebunden.
    // Das heißt, 'this' bezieht sich auf den Kontext, in dem die Arrow Function definiert wurde.
    // Der Kontext hier ist der globale Kontext (oder undefined im strict mode), nicht 'obj'.
    // Ausgabe: 10
  }
};

console.log(obj.regularFunction()) // 42
console.log(obj.arrowFunction()) // 10


// Noch klarer wird es, wenn ich folgendes mache:

const obj2 = {
  value: 100,
  regularFunction: obj.regularFunction, // Achtung, hier wird nicht aufgerufen, sondern nur die Referenz übergeben
  arrowFunction: obj.arrowFunction // Achtung, hier wird nicht aufgerufen, sondern nur die Referenz übergeben
};

console.log(obj2.regularFunction()) // 100, da 'this' jetzt auf 'obj2' zeigt, der Kontext wird dynamisch durch den Aufruf bestimmt
console.log(obj2.arrowFunction()) // immernoch 10, da 'this' immer noch auf den ursprünglichen Kontext zeigt, in dem die Arrow Function definiert wurde (global)

`} />
:::

:::info
**Wann wird welche Art, eine Funktion zu definieren, verwenden?**

* Verwende reguläre Funktionen, wenn du ein eigenes `this` benötigst (z.B. in Methoden von Objekten).
* Verwende Arrow Functions für kürzere Syntax und wenn du das `this` des umgebenden Kontexts beibehalten möchtest (z.B. in Callbacks).

:::

### Gültigkeitsbereiche & Closures

* Gültigkeitsbereiche (Scopes) bestimmen die Sichtbarkeit von Variablen und Funktionen.
    * Globaler Scope: Variablen sind überall im Code zugänglich.
    * Funktionaler Scope: Variablen sind nur innerhalb der Funktion zugänglich.
    * Block-Scope: Mit `let` und `const` deklarierte Variablen sind nur innerhalb des Blocks (z.B. in Schleifen oder if-Anweisungen) zugänglich.

* Hoisting
    * Variablen und Funktions**deklarationen** werden an den Anfang ihres Gültigkeitsbereichs "gehoben".
    * `var`-Variablen werden initialisiert mit `undefined`, während `let` und `const` nicht initialisiert werden und einen Fehler verursachen, wenn sie vor ihrer Deklaration verwendet werden.
        * Das bedeutet, dass `var`-Variablen vor ihrer Deklaration existieren, aber `undefined` sind, was ziemlich gefährlich sein kann
    * Funktionsdeklarationen werden vollständig gehoben, sodass sie vor ihrer Definition aufgerufen werden können.

* Closure
    * Funktionen haben Zugriff auf Variablen aus ihrem äußeren Gültigkeitsbereich, auch nachdem dieser Gültigkeitsbereich beendet ist.
    * Man könnte sagen, Funktionen "merken" sich die Umgebung, in der sie erstellt wurden

<LiveJavascript
    initialCode={`
function outer() {
  let count = 0;
  return function inner() {
    count++;
    return count;
  };
}
const countAndReturn = outer();
console.log(countAndReturn()); // 1
console.log(countAndReturn()); // 2
console.log(countAndReturn()); // 3
`} />

### Klassen & prototypenbasierte Vererbung

#### Grundlagen der Objekte in JavaScript

* Alles in JavaScript ist ein Objekt oder kann als Objekt behandelt werden.
* Objekte haben Eigenschaften (Properties) und Methoden (Funktionen, die als Eigenschaften eines Objekts definiert sind).
* Jedes Objekt hat ein **Prototyp-Objekt**, von dem es erbt
    * Prototypenbasierte Vererbung bedeutet, dass Objekte Eigenschaften und Methoden von ihrem Prototypen erben können.
    * JavaScript hat also keine klassische Klassenhierarchie wie in Java
    * Stattdessen wird die Vererbung durch Prototypenketten realisiert.
* Erzeugung eines Objekts mit Konstrukturfunktion:

:::info
Spoiler: Das werdet ihr in der Praxis nie verwenden und ich werde euch nie danach fragen
:::

<LiveJavascript
    initialCode={`
function Person(name) {
  this.name = name;
}

Person.prototype.greet = function() {
  console.log("Hallo, ich bin " + this.name);
};

const alice = new Person("Alice");
alice.greet(); // "Hallo, ich bin Alice"
`} />

* Erklärung:
    * alice erbt Methoden von Person.prototype
    * Zugriff: JS durchsucht zuerst das eigene Objekt und geht dann nach und nach die Prototypenkette runter.

#### ES6-Klassen-Syntax (syntactic sugar)

:::info
Das hier widerum solltet ihr euch merken
:::

* Seit ES6 (ECMAScript 2015) gibt es eine Klassen-Syntax, die das Arbeiten mit Objekten und Vererbung erleichtert.
* Klassen nur Syntaxzucker über dem bestehenden Prototypensystem.

<LiveJavascript
    initialCode={`
class Person {
  constructor(name) {
    this.name = name;
  }

  greet() {
    console.log("Hallo, ich bin " + this.name);
  }
}

const bob = new Person("Bob");
bob.greet(); // "Hallo, ich bin Bob"
`} />

* Methoden, die in einer Klasse definiert sind, werden automatisch zu dem unterliegenden Prototypen des konstruierten Objekts hinzugefügt.
    * Das bedeutet, die Objekte, die ich über den Konstruktur erzeuge, haben nicht selbst die Methode `greet`, sondern deren Prototyp.
    * Wenn ich also 10.000 Objekte der Klasse `Person` erzeuge, dann gibt es nur eine einzige Kopie der Methode `greet` im Speicher, die von allen Instanzen geteilt wird. Das spart Speicherplatz und verbessert die Leistung.
    * Wenn ich `greet` als normale Funktion innerhalb des Konstruktors definiert hätte (z.B. `this.greet = function() { ... }`), dann hätte jede Instanz ihre eigene Kopie der Methode, was ineffizient wäre.
    * Ich könnte theoretisch auch Methoden des Prototyps an einem Objekt überschreiben, indem ich einfach eine Methode mit dem gleichen Namen direkt auf dem Objekt definiere (z.B. `bob.greet = function() { ... }`), aber das ist selten sinnvoll.
    * Wenn Instanzen von Klassen im Browser in den Dev Tools inspected werden, sehen sie aus wie normale Objekte, aber sie haben eine versteckte `[[Prototype]]` (oder `__prototype__`)-Eigenschaft, die auf das Prototyp-Objekt der Klasse zeigt.
        * In dieser Property sind die Methoden und Eigenschaften definiert, die von der Klasse bereitgestellt werden.

* Klassen können von anderen Klassen mit dem `extends`-Schlüsselwort erben.

```javascript

class Student extends Person {
  constructor(name, courseOfStudy) {
    super(name); // ruft Konstruktor der Elternklasse (des Prototyps) auf
    this.courseOfStudy = courseOfStudy;
  }
  informAboutCourseOfStudy() {
    console.log(`${this.name} studiert ${this.courseOfStudy}`);
  }
}

const clara = new Student("Clara", "Informatik");
clara.greet(); // geerbt von Person
clara.informAboutCourseOfStudy(); // eigene Methode

```

* Klassen können auch statische Methoden haben, die auf der Klasse selbst und nicht auf den Instanzen der Klasse aufgerufen werden.

```javascript
class Math {
  static add(a, b) {
    return a + b;
  }
}
console.log(Math.add(5, 3)); // 8
```

* Klassen können auch getter und setter Methoden haben, um den Zugriff auf Eigenschaften zu kontrollieren.

```javascript
class Kreis {
  constructor(radius) {
    this.radius = radius;
  }
  get umfang() {
    return 2 * Math.PI * this.radius;
  }

  set umfang(value) {
      this.radius = value / (2 * Math.PI);
  }
}

const kreis = new Kreis(5);
console.log(kreis.umfang); // Zugriff wie auf eine Eigenschaft, aber es wird die Methode ausgeführt
```

#### Welche Konsequenzen ergeben sich aus der prototypische Vererbung?

* In Javascript ist `class` nur eine "schönere Schreibweise" für Prototypen
    * Konsequenz: Entwickler:innen müssen verstehen, dass es keine **echten Klassen** gibt, sondern nur Objekte, die von anderen Objekten erben.
    * Praktischer Unterschied: Man kann nachträglich **Objekte erweitern oder verändern**, was in klassischen OOP-Sprachen nicht oder nur sehr eingeschränkt möglich ist.
    * Beispiel:

```javascript
Person.prototype.sayBye = function() {
  console.log("Tschüss!");
};
// fügt nachträglich allen Person-Instanzen eine Methode hinzu
```

* Flexibilität vs. Strenge
    * In klassischen OOP-Sprachen sind Klassen und ihre Beziehungen oft strikt definiert; Es gibt eine starre Struktur, einen klaren Bauplan.
    * In JS: Man kann Objekte jederzeit dynamisch verändern und erweitern
    * Konkrete Folge:
        * **Mehr Freiheit**, aber auch **Fehlerpotenzial** (z.B. können Methoden überschrieben werden, ohne dass die Entwickler:in dies beabsichtigt hat)
        * Es gibt keinen echten Typenzwang, keine starke Typisierung; Man prüft eher auf Fähigkeit (Duck Typing) als auf die Zugehörigkeit zu einem Typ selbst.

```javascript
if (obj.methode) {
...
}
```

* Durch die Prototypen Kette kann es manchmal verwirrend sein, woher eine Methode oder Eigenschaft tatsächlich stammt.
    * Entwickler:innen müssen verstehen, wie die Prototypen-Kette funktioniert, um Fehler zu vermeiden.

```javascript
const obj = {};
console.log(obj.toString()); // Funktioniert, weil obj von Object.prototype erbt
```

## 2. Event Handling

### Was sind Events

* Events sind Aktionen oder Vorkommnisse, die im Browser auftreten und auf die per Javascript reagiert werden kann
* Beispiele für Events:
    * Benutzerinteraktionen: Klicks, Tastendrücke, Mausbewegungen, Touch-Gesten
    * Ladeereignisse: Seite geladen, Bild geladen
    * Formularereignisse: Formular abgeschickt, Eingabefeld geändert
    * Fensterereignisse: Fenstergröße geändert, Seite verlassen
* Auf Events kann mit sogenannten Event-Listenern und Callback-Funktionen reagiert werden
* Eine Callback-Funktion ist eine Funktion, die als Argument an eine andere Funktion übergeben wird und später aufgerufen wird, wenn das Event eintritt

### Hinzufügen von Event-Listenern

* Event-Listener können auf verschiedene Arten hinzugefügt werden:

```html
<!-- ❌ Inline im HTML (nicht empfohlen, veraltet) -->
<button onclick="alert('Button geklickt!')">Klick mich</button>
```

```javascript
// ❌ Mit der on<Event>-Eigentschaft eines DOM-Elements (veraltet)
const button = document.getElementById('myButton');
button.onclick = function() {
  alert('Button geklickt!');
};

```

```javascript
// ✅ Mit der `addEventListener`-Methode (empfohlen)
const button = document.getElementById('myButton');
button.addEventListener('click', function() {
  alert('Button geklickt!');
});
```

### Häufige Eventtypen

* Maus: `click`, `dblclick`, `mouseover`, `mouseout`, `mousemove`, `mousedown`, `mouseup`
* Tastatur: `keydown`, `keyup`, `keypress`
* Formular: `submit`, `change`, `input`, `focus`, `blur`
* Fenster/DOM: `load`, `resize`, `scroll`, `unload`, `DOMContentLoaded`
* Touch: `touchstart`, `touchmove`, `touchend`

### Event-Objekt

* Jedes Event, das ausgelöst wird, erzeugt ein Event-Objekt, das Informationen über das Event enthält
* Das Event-Objekt wird automatisch an die Callback-Funktion übergeben

```javascript
button.addEventListener('click', function(event) {
  console.log('Button geklickt bei Position:', event.clientX, event.clientY);
});
```

* Wichtige Eigenschaften des Event-Objekts:
    * `type`: Der Typ des Events (z.B. 'click', 'keydown')
    * `target`: Das DOM-Element, auf dem das Event ausgelöst wurde
    * `currentTarget`: Das DOM-Element, an das der Event-Listener gebunden ist
    * `preventDefault()`: Methode, um die Standardaktion des Events zu verhindern (z.B. das Absenden eines Formulars)
    * `stopPropagation()`: Methode, um die Weiterleitung des Events in der DOM-Hierarchie zu stoppen
    * Für Forms: `event.target.elements` enthält alle Formularelemente

### Event Flow (Capturing & Bubbling)

* Events durchlaufen drei Phasen:
    1. Capturing Phase: Das Event wird von der Wurzel des DOM-Baums bis zum Ziel-Element weitergeleitet
    2. Target Phase: Das Event erreicht das Ziel-Element
    3. Bubbling Phase: Das Event wird vom Ziel-Element zurück zur Wurzel des DOM-Baums weitergeleitet

```javascript
document.body.addEventListener("click", () => console.log("Body!"));
document.querySelector("button").addEventListener("click", () => console.log("Button!"));
// Klick auf Button → "Button!" dann "Body!"
```

### Event Delegation

* Event Delegation ist ein Muster, bei dem ein einzelner Event-Listener auf einem übergeordneten Element verwendet wird, um Events von untergeordneten Elementen zu behandeln
* Vorteil: Spart Speicher und verbessert die Leistung, besonders bei vielen ähnlichen Elementen

```javascript
document.getElementById('parent').addEventListener('click', function(event) {
  if (event.target && event.target.matches('button.child')) {
    console.log('Child Button geklickt:', event.target);
  }
});
```

## 3. DOM Manipulation

* Recap: Was ist das DOM?
    * Das Document Object Model (DOM) ist eine Programmierschnittstelle für HTML- und XML-Dokumente.
    * Repräsentiert die HTML-Struktur als Baum von Objekten
    * jeder Knoten = HTML-Element, Attribut oder Text
* DOM Manipulation bezieht sich auf das Ändern der Struktur, des Stils oder des Inhalts eines Dokuments über JavaScript

### DOM-Elemente auswählen

* Methoden zum Auswählen von DOM-Elementen:
    * `getElementById(id)`: Wählt ein Element anhand seiner ID aus
    * `getElementsByClassName(className)`: Wählt alle Elemente mit einer bestimmten Klasse aus (HTMLCollection)
    * `getElementsByTagName(tagName)`: Wählt alle Elemente mit einem bestimmten Tag-Namen aus (HTMLCollection)
    * `querySelector(selector)`: Wählt das erste Element aus, das dem CSS-Selektor entspricht
    * `querySelectorAll(selector)`: Wählt alle Elemente aus, die dem CSS-Selektor entsprechen (NodeList)

```javascript
const elementById = document.getElementById('myId');
const elementsByClass = document.getElementsByClassName('myClass');
const elementsByTag = document.getElementsByTagName('div');
const firstElement = document.querySelector('.myClass');
const allElements = document.querySelectorAll('.myClass');
```

* Auf verwandte Elemente zugreifen:
    * `parentNode`: Gibt das übergeordnete Element zurück
    * `childNodes`: Gibt eine NodeList aller Kindelemente zurück (inkl. Textknoten)
    * `children`: Gibt eine HTMLCollection aller Kindelemente zurück (nur Elementknoten)
    * `firstChild` / `lastChild`: Gibt das erste/letzte Kindelement zurück (inkl. Textknoten)
    * `firstElementChild` / `lastElementChild`: Gibt das erste/letzte Kindelement zurück (nur Elementknoten)
    * `nextSibling` / `previousSibling`: Gibt das nächste/vorherige Geschwisterelement zurück (inkl. Textknoten)
    * `nextElementSibling` / `previousElementSibling`: Gibt das nächste/vorherige Geschwisterelement zurück (nur Elementknoten)

```javascript
const elementById = document.getElementById('myId');
const parent = elementById.parentNode;
const firstChild = elementById.firstElementChild;
//...
```

    * `removeChild`: Entfernt ein Kindelement
    * `append`: Fügt Kindelemente oder Text hinzu (moderne Methode)
    * `prepend`: Fügt Kindelemente oder Text am Anfang hinzu (moderne Methode)

    * `after`: Fügt Kindelemente oder Text nach dem Element hinzu (moderne Methode)

* Attribute:

    * `removeAttribute(name)`: Entfernt ein Attribut eines Elements

## 4. Validierung von Formularen

### Warum sollten Formulare validiert werden?

* Validierung stellt sicher, dass die Daten, die von Benutzern eingegeben werden, korrekt und vollständig sind, bevor sie verarbeitet und an den Server gesendet werden.
* Dies ersetzt nicht die serverseitige Validierung, sondern ergänzt sie.
* Durch clientseitige Validierung wird die Benutzererfahrung verbessert, da Fehler sofort angezeigt werden können, ohne dass eine Serveranfrage erforderlich ist.
* Außerdem werden unnötige Serveranfragen vermieden, was die Leistung der Anwendung verbessert.

### HTML5-Basisvalidierung

* HTML5 bietet eingebaute Validierungsfunktionen für Formularelemente.
* Wichtige Attribute:
    * `required`: Macht ein Feld obligatorisch
    * `type`: Definiert den Typ des Eingabefelds (z.B. `email`, `number`, `url`, `tel`, `date`)
    * `min` und `max`: Definieren den minimalen und maximalen Wert für numerische Eingaben
    * `minlength` und `maxlength`: Definieren die minimale und maximale Länge für Textfelder
    * `pattern`: Definiert ein reguläres Ausdrucksmuster, das die Eingabe erfüllen muss
* Der Browser führt die Validierung automatisch durch, wenn das Formular abgesendet wird.

```html
<input type="email" required>
```

### Clientseitige Validierung mit JavaScript

* JavaScript kann verwendet werden, um benutzerdefinierte Validierungsregeln zu implementieren und sofortiges Feedback zu geben.
* Dadurch kann die Validierung an spezifische Anforderungen angepasst werden, bietet erheblich mehr Kontrolle und individuelles Feedback.


```javascript
const form = document.getElementById('myForm');
form.addEventListener('submit', function(event) {
  const email = form.elements['email'].value;
  if (!validateEmail(email)) {
    event.preventDefault(); // Verhindert das Absenden des Formulars
    alert('Bitte geben Sie eine gültige E-Mail-Adresse ein.');
  }
});
```

### Serverseitige Validierung

* Serverseitige Validierung wird hier nicht behandelt
* Es ist aber zwingend notwendig, darauf hinzuweisen, dass clientseitige Validierung niemals die serverseitige Validierung ersetzen kann
* Wenn Daten an den Server gesendet werden, müssen sie immer auf dem Server validiert werden, um Sicherheit und Datenintegrität zu gewährleisten.

## 5. Zustand speichern

### Warum Zustand speichern?

* Webanwendungen sind von Natur aus zustandslos (stateless), da HTTP ein zustandsloses Protokoll ist.
* Das bedeutet, dass jede Anfrage unabhängig von vorherigen Anfragen behandelt wird.
* Um eine konsistente Benutzererfahrung zu gewährleisten, müssen Webanwendungen den Zustand speichern, z.B. Benutzereinstellungen, Warenkörbe, Anmeldestatus usw.
* Es gibt verschiedene Techniken, um den Zustand in Webanwendungen zu speichern, sowohl auf der Client- als auch auf der Serverseite.
* An dieser Stelle konzentrieren wir uns auf clientseitige Speichertechniken.

### Globale Variablen

* Globale Variablen sind Variablen, die im globalen Scope definiert sind und von überall im Skript zugänglich sind.
* Sie sind einfach zu verwenden, aber können leicht zu Namenskonflikten und schwer wartbarem Code führen.
* Sie sind nur für die Dauer der Sitzung verfügbar (gehen verloren, wenn die Seite neu geladen wird).
* Einfach umzusetzen, aber nicht persistent.

```javascript
let globalState = {
  isLoggedIn: false,
  cartItems: []
};

function login() {
  globalState.isLoggedIn = true;
}
```

### Cookies

* Cookies sind kleine Textdateien, die im Browser gespeichert werden und Daten zwischen Sitzungen speichern können.
* Sie werden automatisch mit HTTP-Headern gesendet und können (sowohl auf der Client- als auch) auf der Serverseite gelesen und geschrieben werden.
* Cookies haben Einschränkungen hinsichtlich Größe (ca. 4KB) und Anzahl (ca. 20 pro Domain).

```javascript
document.cookie = "username=JohnDoe; expires=Fri, 31 Dec 2024 23:59:59 GMT; path=/";
```

### Web Storage API

* Die Web Storage API bietet zwei Mechanismen zum Speichern von Daten im Browser: `localStorage` und `sessionStorage`.
* `localStorage`: Key-Value-Speicher im Browser, der Daten dauerhaft speichert (bis sie gelöscht werden).
    * Daten bleiben auch nach dem Schließen des Browsers erhalten.
    * Einfacher Zugriff per API (`setItem`, `getItem`, `removeItem`, `clear`).
    * Nur Strings können gespeichert werden (andere Datentypen müssen serialisiert werden, z.B. mit JSON).
    * Wird *nicht* automatisch mit HTTP-Headern gesendet (im Gegensatz zu Cookies).
    * Daten sind auf die gleiche Domain beschränkt (Same-Origin-Policy).

```javascript
localStorage.setItem('username', 'JohnDoe');
const username = localStorage.getItem('username');
localStorage.removeItem('username');
localStorage.clear();
```

* `sessionStorage`: Key-Value-Speicher im Browser, der Daten nur für die Dauer der Sitzung speichert (gehen verloren, wenn der Tab oder Browser geschlossen wird).
    * Ansonsten gleiche API und Einschränkungen wie `localStorage`.

```javascript
sessionStorage.setItem('sessionId', 'abc123');
const sessionId = sessionStorage.getItem('sessionId');
sessionStorage.removeItem('sessionId');
sessionStorage.clear();
```

### IndexedDB
* IndexedDB ist eine Low-Level-API für die Speicherung großer Mengen strukturierter Daten im Browser.
* Quasi eine NoSQL-Datenbank im Browser.
* Sie ermöglicht das Speichern von Objekten und bietet leistungsstarke Such- und Indexierungsfunktionen.
* Komplexer zu verwenden als `localStorage` und `sessionStorage`, aber auch viel leistungsfähiger.
* Unterstützt Transaktionen und asynchrone Operationen über Promises oder Events, blockiert also nicht den UI-Thread.
* Daten sind auf die gleiche Domain beschränkt (Same-Origin-Policy).
* Die Daten sind persistent und bleiben auch nach dem Schließen des Browsers erhalten.
* Einsatzszenarien:
    * Offline-First-Webapps (z.B. Progressive Web Apps)
    * Caching von großen Datenmengen (z.B. Mediendateien, umfangreiche JSON-Daten)
    * Anwendungen mit vielen strukturierten Datensätzen (z.B. Notizen-Apps, To-Do-Listen)

```javascript
// 1. Datenbank öffnen oder erstellen
// - Name: "MeineDatenbank"
// - Version: 1 (wird bei Änderungen hochgezählt)
const request = indexedDB.open("MeineDatenbank", 1);

// 2. Event: Wenn eine neue Version erstellt wird oder DB noch nicht existiert
request.onupgradeneeded = e => {
  const db = e.target.result;

  // Überprüfen, ob es den Object Store "users" schon gibt
  // Falls nicht, wird er angelegt
  if (!db.objectStoreNames.contains("users")) {
    // keyPath = "id" → jedes Objekt braucht ein eindeutiges Feld "id"
    db.createObjectStore("users", { keyPath: "id" });
  }
};

// 3. Event: Wenn die Datenbank erfolgreich geöffnet wurde
request.onsuccess = e => {
  const db = e.target.result; // Datenbank-Objekt

  // Eine Transaktion starten
  // - erster Parameter: Name des Object Stores
  // - zweiter Parameter: "readwrite" → wir wollen schreiben
  const tx = db.transaction("users", "readwrite");

  // Zugriff auf den Object Store "users"
  const store = tx.objectStore("users");

  // Ein Objekt speichern (oder überschreiben, falls id schon existiert)
  const user = { id: 1, name: "Alice", email: "alice@example.com" };
  store.put(user);

  // Optional: Noch ein Objekt speichern
  store.put({ id: 2, name: "Bob", email: "bob@example.com" });

  // Wenn Transaktion abgeschlossen ist
  tx.oncomplete = () => {
    console.log("Alle Daten erfolgreich gespeichert!");
  };

  // Fehlerbehandlung in der Transaktion
  tx.onerror = () => {
    console.error("Fehler beim Speichern:", tx.error);
  };

  // Jetzt können wir auch Daten abrufen:
  const readTx = db.transaction("users", "readonly");
  const readStore = readTx.objectStore("users");

  // Suche nach Nutzer mit id = 1
  const getRequest = readStore.get(1);

  getRequest.onsuccess = () => {
    console.log("Gefundener Nutzer:", getRequest.result);
    // Ausgabe: { id: 1, name: "Alice", email: "alice@example.com" }
  };
};
```

### Zusammenfassung und Vergleich der Techniken

| Technik          | Persistenz           | Kapazität        | API-Komplexität | Anwendungsfälle                          |
|------------------|----------------------|------------------|-----------------|------------------------------------------|
| Globale Variablen| Nur während Sitzung  | Sehr gering      | Sehr einfach    | Temporäre Zustände während der Sitzung   |
| Cookies          | Bis Ablaufdatum      | Ca. 4KB pro Cookie| Einfach         | Sitzungsmanagement, Tracking             |
| localStorage     | Dauerhaft            | Ca. 5-10MB       | Einfach         | Benutzereinstellungen, Caching           |
| sessionStorage   | Bis Tab/Browser geschlossen | Ca. 5-10MB | Einfach         | Temporäre Daten während einer Sitzung    |
| IndexedDB        | Dauerhaft            | Sehr groß (100MB+) | Komplex         | Offline-Apps, große strukturierte Daten  |

## 6. Asynchrone Programmierung

### Warum asynchrone Programmierung?

* **JavaScript ist Single-Threaded**: Es gibt immer nur einen Ausführungsstrang
* Ohne Asynchronität würde eine lange Operation (z.B. API-Call, Daten lesen) die komplette Seite blockieren
* Asynchronität = Aufgaben werden im Hintergrund ausgeführt und das Ergebnis wird später verarbeitet
* Beispiele:
    * API-Calls
    * Timer (setTimeout, setInterval)
    * Benutzer-Events (Klicks, Tastendrücke)

### Event Loop & Callbacks

* Event Loop = Mechanismus, der Events, Tasks und Promises abarbeitet
* Callbacks sind Funktionen, die später aufgerufen werden, sobald ein Ergebnis vorliegt

```javascript
console.log("Start");

setTimeout(() => {
  console.log("Timer fertig!");
}, 1000);

console.log("Ende");
// Ausgabe: Start → Ende → Timer fertig!
```

### Callbacks & Callback Hell

* Früher wurden Callbacks verwendet, um asynchrone Operationen zu handhaben
* Problem: Callback Hell (verschachtelte Callbacks, schwer lesbar)

```javascript
loadUser(id, user => {
  loadPosts(user, posts => {
    loadComments(posts, comments => {
      console.log(comments);
    });
  });
});
```

### Promises

* Promises sind Objekte, die einen zukünftigen Wert repräsentieren, der jetzt, später oder nie verfügbar ist
* Promises lösen das Problem der Callback Hell
* Immer wenn auf einen Promise gewartet wird, kann eine andere Aufgabe im Event Loop ausgeführt werden
* Drei Zustände: pending, fulfilled, rejected
* Methoden: `then`, `catch`, `finally`

```javascript
fetch("https://jsonplaceholder.typicode.com/posts/1")
  .then(response => response.json())
  .then(data => console.log(data))
  .catch(err => console.error("Fehler:", err))
  .finally(() => console.log("Fertig"));
```

### Async/Await (ES2017)

* Async/Await ist eine moderne Syntax um mit Promises zu arbeiten
* Durch die Nutzung von Async/Await wird asynchroner Code wie synchroner Code lesbar
* `async` markiert eine Funktion als asynchrone Funktion, die einen Promise zurückgibt
* `await` wartet auf die Erfüllung des Promises
* `void` kann verwendet werden, um eine async Funktion aufzurufen, ohne auf das Ergebnis zu warten
* In einer async Funktion kann und sollte `try/catch` für Fehlerbehandlung verwendet werden
* `finally` wird immer ausgeführt, egal ob ein Fehler aufgetreten ist oder nicht

:::info
Moderne Applikationen sollten bevorzugt Async/Await verwenden
:::


```javascript

async function loadPost() {
  try {
    const response = await fetch("https://jsonplaceholder.typicode.com/posts/1");
    const data = await response.json();
    console.log(data);
  } catch (err) {
    console.error("Fehler:", err);
  } finally {
    console.log("Fertig");
  }
}
void loadPost();
```

### Mehrere Promises gleichzeitig verarbeiten

* `Promise.all`: Wartet, bis alle Promises erfüllt sind, oder lehnt ab, wenn eines fehlschlägt
* `Promise.race`: Wartet, bis das erste Promise erfüllt oder abgelehnt wird
* `Promise.allSettled`: Wartet, bis alle Promises entweder erfüllt oder abgelehnt sind, und gibt deren Ergebnisse zurück

```javascript
const promise1 = fetch("https://jsonplaceholder.typicode.com/posts/1").then(res => res.json());
const promise2 = fetch("https://jsonplaceholder.typicode.com/posts/2").then(res => res.json());
try {
    const results = await Promise.all([promise1, promise2]);
    console.log("Beide Promises erfüllt:", results);
} catch (err) {
    console.error("Eines der Promises ist fehlgeschlagen:", err);
}
```

## 7. Kommunikation zwischen Client & Server

### Gängige Datenformate

#### JSON (JavaScript Object Notation)

* Leichtgewichtiges Datenformat, das menschenlesbar und relativ einfach zu parsen ist
* Recht wenig Overhead, enthält aber trotzdem Strukturinformationen
* Wird häufig für den Datenaustausch zwischen Client und Server verwendet
* Heutzutage das bevorzugte Datenübertragungsformat für APIs und Webservices
* Beispiel:

```json
{
  "name": "Alice",
  "age": 30,
  "isStudent": false,
  "courses": ["Math", "Science"],
  "address": {
    "street": "123 Main St",
    "city": "Wonderland"
  }
}
```

* In JavaScript kann JSON einfach mit `JSON.stringify()` in einen String umgewandelt und mit `JSON.parse()` wieder in ein Objekt zurückverwandelt werden.

<LiveJavascript

#### XML (eXtensible Markup Language)

* Älteres, aber immer noch verwendetes Datenformat
* Eher komplex und heutzutage weniger gebräuchlich als JSON
* Bietet mehr Flexibilität und Erweiterbarkeit, aber auf Kosten von Lesbarkeit und Einfachheit
* Hat mehr Overhead durch Tags und Attribute
* Beispiel:

```xml
<person>

  <address>
    <street>123 Main St</street>
    <city>Wonderland</city>

```

* In JavaScript kann XML mit dem `DOMParser` geparst und mit `XMLSerializer` wieder in einen String umgewandelt werden.

#### YAML (YAML Ain't Markup Language/Yet Another Markup Language)

* Ein weiteres Datenformat, das menschenlesbar und einfach zu schreiben ist
* Wird häufig für Konfigurationsdateien verwendet
* Weniger gebräuchlich für den Datenaustausch zwischen Client und Server
* Größter Nachteil (in meinen Augen): Wer es ernsthaft für eine gute Idee gehalten hat, nicht sichtbaren Zeichen eine semantische Bedeutung zu geben, gehört gesteinigt.
    * Einrückungen sind bedeutungsvoll (ähnlich wie in Python)
    * Einrückung bedeutet eine Hierarchieebene

```yaml
  - Math
  - Science

    street: 123 Main St
    city: Wonderland
```

* In JavaScript kann YAML mit Bibliotheken wie `js-yaml` geparst und serialisiert werden.

### Networking APIs

#### XMLHttpRequest (XHR)

* Ältere Methode, um HTTP-Anfragen zu machen
* Wird immer noch in vielen Legacy-Anwendungen verwendet
* Unterstützt von allen Browsern, auch älteren
* Komplexere API im Vergleich zu moderneren Methoden
* Beispiel:

<LiveJavascript
    initialCode={`
try {
    const response = await new Promise((resolve, reject) => {
      const xhr = new XMLHttpRequest();
      xhr.open("GET", "https://jsonplaceholder.typicode.com/posts/1");
      xhr.onload = function() {
        if (xhr.status >= 200 && xhr.status < 300) {

        reject(new Error("Netzwerkfehler"));
      };

} catch (err) {

}
`} />

#### fetch API

* Moderne Methode, um HTTP-Anfragen zu machen
* Ersetzt ältere Methoden wie `XMLHttpRequest`
* wird von allen modernen Browsern unterstützt

<LiveJavascript
    initialCode={`

* Die fetch-API ist Promise basiert
* Standardmäßig werden nur Netzwerkfehler als Fehler betrachtet (z.B. kein Internet)
* Die Antwort enthält immer ein `Response`-Objekt, auch bei HTTP-Fehlern (z.B. 404, 500)
* Um HTTP-Fehler zu behandeln, muss der Statuscode manuell überprüft werden

<LiveJavascript

#### Vergleich von fetch API und XHR


| Kategorie          | Fetch API           | XHR |
|--------------------|---------------------|-----|
| Syntax             | Modern, Promise-basiert | Veraltet, Callback-basiert |
| Fehlerbehandlung   | Manuell (Statuscode prüfen) | Automatisch (HTTP-Fehler lösen Fehler aus) |
| Unterstützung      | Alle modernen Browser | Alle Browser, auch ältere |
| Streaming          | Unterstützt (Response.body) | Nicht unterstützt |
| Einfachheit       | Einfacher zu lesen und zu schreiben | Komplexer, mehr Boilerplate-Code |

### AJAX (Asynchronous JavaScript and XML)

* AJAX ist keine spezielle Technik, sondern ein Konzept, das es ermöglicht, Daten asynchron im Hintergrund zu laden, ohne die Seite neu zu laden
* Das XML im Namen ist historisch bedingt; heutzutage wird meistens JSON verwendet

* Vor AJAX musste für jede einzelne Änderung von Daten im Backend die gesamte Seite neu geladen werden
* Mit AJAX können Teile der Seite aktualisiert werden, ohne die gesamte Seite neu zu laden
* beispielhafte Anwendungsfälle:
    * Dynamisches Nachladen von Inhalten (z.B. Social Media Feeds, Infinite Scroll)
    * Formulare absenden, ohne die Seite zu verlassen
    * Chat Anwendungen

* AJAX kombiniert verschiedene Technologien
    * Javascript: steuert die Logik
    * DOM: zeigt die Daten an
    * HTTP-Requests: holen Daten vom Server
    * XMLHttpRequest oder fetch API: führen die HTTP-Requests aus
    * JSON/XML: Datenformat für den Datenaustausch

## Zusammenfassung

In diesem Kapitel hast du gelernt, wie JavaScript Webseiten zum Leben erweckt. Hier sind die wichtigsten Takeaways:

* ✅ **JavaScript ist die Programmiersprache des Webs** - Sie ermöglicht Interaktivität und dynamisches Verhalten im Browser
* ✅ **Event Handling** macht Webseiten interaktiv - Event-Listener reagieren auf Benutzeraktionen wie Klicks, Tastendrücke oder Formulareingaben
* ✅ **DOM Manipulation** verändert Inhalte dynamisch - JavaScript kann HTML-Struktur, Inhalte und Styles zur Laufzeit anpassen
* ✅ **Asynchrone Programmierung** ist essentiell - Async/Await und Promises ermöglichen es, Aufgaben im Hintergrund auszuführen ohne die Seite zu blockieren
* ✅ **Client-Server-Kommunikation** mit der fetch API - AJAX ermöglicht das Laden und Senden von Daten ohne Neuladen der Seite
* ✅ **Zustand speichern** mit localStorage, sessionStorage oder IndexedDB - Clientseitige Speicherlösungen verbessern die User Experience
* ✅ **Validierung** gehört sowohl auf Client- als auch auf Serverseite - Clientseitige Validierung verbessert UX, serverseitige Validierung garantiert Sicherheit

Auch wenn in der Praxis nur noch selten mit purem HTML, CSS und JavaScript gearbeitet wird, sind die hier behandelten Konzepte die Grundlage für moderne Frameworks und Bibliotheken wie React, Angular oder Vue.js.
Ein solides Verständnis dieser Grundlagen ist entscheidend, um komplexe Webanwendungen erfolgreich zu entwickeln.
