## Motivation

* JavaScript ist eine dynamisch typisierte Sprache
* Dynamische Typisierung bedeutet, dass Variablen zur Laufzeit ihren Typ ändern können
* Vorteile:
    * Schnell und flexibel
    * Weniger Boilerplate-Code
* Nachteile:
    * Fehler erst zur Laufzeit sichtbar
    * Fehler schwerer zu finden
    * Schwieriger zu warten
    * Weniger Unterstützung durch IDEs (Autovervollständigung, Refactoring, etc.)
* TypeScript ist eine statisch typisierte Obermenge von JavaScript
* TypeScript erweitert JavaScript um statische Typen und andere Features
* TypeScript wird zu JavaScript transpiliert, das dann in Node.js ausgeführt werden kann
* Vorteile:
    * Fehler werden bereits zur Entwicklungszeit erkannt
    * Bessere Unterstützung durch IDEs
    * Besser wartbarer Code
* Nachteile:
    * Etwas mehr Boilerplate-Code
    * Lernkurve für Entwickler, die nur JavaScript kennen
    * Transpilationsschritt notwendig
* TypeScript ist mittlerweile weit verbreitet und wird von vielen großen Projekten und Unternehmen genutzt (z.B. Angular, Microsoft, Slack, etc.)
* TypeScript ist eine gute Wahl für größere Projekte und Teams, die Wert auf Wartbarkeit und Codequalität legen
* Für kleine Projekte oder Prototypen kann JavaScript ausreichend sein
* In dieser Vorlesung werden wir TypeScript für die Backend-Entwicklung mit Node.js nutzen
* Wir werden die Grundlagen von TypeScript kennenlernen und wie es in einem Node.js Projekt eingesetzt werden kann
* Wir werden auch sehen, wie TypeScript mit gängigen Node.js Modulen und Bibliotheken zusammenarbeitet
* Ziel: Sauberen, wartbaren und gut typisierten Code schreiben, der die Vorteile von TypeScript nutzt

## Praxisbeispiel, wie TypeScript in einem Node.js Projekt unterstützen kann

* Man stelle sich folgende Funktion vor:

```javascript
function add(a, b) {
    return a + b;
}
```

* Auf den ersten Blick sieht die Funktion harmlos aus
* Doch was passiert, wenn die Funktion mit unterschiedlichen Typen aufgerufen wird?

<LiveJavascript
initialCode={`
function add(a, b) {
    return a + b;
}
console.log(add(1, 2)); // 3
console.log(add("1", "2")); // "12"
console.log(add(1, "2")); // "12"
console.log(add(null, undefined)); // null
console.log(add({}, [])); // "[object Object]"
console.log(add(true, false)); // 1
`} />

* Die Funktion verhält sich je nach Eingabetyp unterschiedlich
* Dies kann zu unerwarteten Ergebnissen und Fehlern führen
* Mit TypeScript können wir die Funktion typisieren, um solche Fehler zu vermeiden

```typescript
function add(a: number, b: number): number {
    return a + b;
}
```

* Nun gibt TypeScript einen Fehler aus, wenn die Funktion mit falschen Typen aufgerufen wird

<LiveJavascript
    language="typescript"
    enableTypeDiagnostics={true}
    initialCode={`
function add(a: number, b: number): number {
  return a + b;
}
console.log(add(1, 2));      // ok
console.log(add("1", "2"));  // Fehler
console.log(add(1, "2"));    // Fehler
console.log(add(null, undefined)); // Fehler
console.log(add({}, []));    // Fehler
console.log(add(true, false)); // Fehler
`}
/>

* Dadurch werden Fehler frühzeitig erkannt und können behoben werden
* Dies erhöht die Codequalität und Wartbarkeit des Projekts
* Wichtig: TypeScript hilft nur bei der Entwicklung
    * TypeScript unterstützt beim Build-Prozess des Projekts, Fehler zu finden
    * Zur Laufzeit wird jedoch pures JavaScript ausgeführt
    * TypeScript ersetzt also keine Input-Validierung, Tests oder andere Maßnahmen zur Fehlervermeidung
    * Dass der Code hierüber also trotzdem ausgeführt werden kann, ist kein Fehler von meinem Code, sondern eine Konsequenz davon, dass TypeScript nur zur Entwicklungszeit wirkt
    * In der Praxis wird sich dies darin äußern, dass TypeScript Projekte nicht transpiliert werden können und man daher nicht in Produktion gehen kann, wenn TypeScript Fehler im Code findet
    * Wobei es jedoch nicht helfen kann:
        * User Input
        * Daten von externen APIs oder DatenbankenA

## Basis Syntax von Typescript

* Das Typing System von TypeScript ist sehr mächtig
* Tatsächlich ist das Typing System von TypeScript Turing-vollständig
* Das bedeutet konkret:
    * Man kann nur mit dem Typing System von TypeScript theoretisch jedes beliebige Programm schreiben, ohne eine einzige Zeile JavaScript zu schreiben.
    * Das Typing System ist also extrem mächtig und komplex
* In der Praxis werden jedoch nur ein Bruchteil der Features genutzt
* Hier sind die wichtigsten Konzepte und Syntax-Elemente von TypeScript

### Primitive Typen

* TypeScript unterstützt primitive Typen wie `number`, `string`, `boolean`, `null`, `undefined`

<LiveJavascript
    language="typescript"
    enableTypeDiagnostics={true}
    initialCode={`
let age: number = 25;
let name: string = "John";
let isStudent: boolean = true;
let nothing: null = null;
let notDefined: undefined = undefined;
`} />

### Komplexe Typen

* TypeScript unterstützt komplexe Typen wie `array`, `tuple`, `enum`, `object`, `function`

<LiveJavascript
    language="typescript"
    enableTypeDiagnostics={true}
    initialCode={`
let numbers: number[] = [1, 2, 3];
let tuple: [string, number] = ["John", 25];
enum Color {
    Red,
    Green,
    Blue
}
let color: Color = Color.Red;
let person: { name: string; age: number } = { name: "John", age: 25 };
let greet: (name: string) => string = function(name: string): string {
    return "Hello, " + name;
};
`} />

### Interfaces, Type Aliases, Classes

* TypeScript unterstützt benutzerdefinierte Typen wie `interface`, `type alias`, `class`
* Diese ermöglichen es, komplexe Datenstrukturen zu definieren und wiederzuverwenden

<LiveJavascript
    language="typescript"
    enableTypeDiagnostics={true}
    initialCode={`
interface Person {
    name: string;
    age: number;
}
const person1: Person = { name: "John", age: 25 };

type ID = string | number;
const userId: ID = "123";

class Student {
    constructor(public name: string, public age: number) {}
}
const student1 = new Student("Jane", 22);
`}/>

### Union Types

* TypeScript unterstützt Union Types (z.B. `string | number`)
* Union Types ermöglichen es, einer Variable mehrere Typen zuzuweisen

<LiveJavascript
    language="typescript"
    enableTypeDiagnostics={true}
    initialCode={`
let id: string | number = "123";
id = 123; // auch ok
id = true; // Fehler
`}/>

### Intersection Types

* TypeScript unterstützt Intersection Types (z.B. `A & B`)
* Intersection Types ermöglichen es, eine Variable zu definieren, die alle Eigenschaften von mehreren Typen hat und daher beide Typen erfüllt
* Dadurch können Typen kombiniert werden

<LiveJavascript
    language="typescript"
    enableTypeDiagnostics={true}
    initialCode={`
interface A {
    a: string;
}

interface B {
    b: number;
}

type C = A & B;

let c: C = { a: "hello", b: 42 };
`}/>

### Literal Types

* TypeScript unterstützt Literal Types (z.B. `"hello"` oder `42`)
* Literal Types sind nützlich, um eine Variable auf einen bestimmten Wert zu beschränken

<LiveJavascript
    language="typescript"
    enableTypeDiagnostics={true}
    initialCode={`
type Direction = "up" | "down" | "left" | "right";
let direction: Direction = "up";
direction = "down"; // auch ok
direction = "left"; // auch ok
direction = "right"; // auch ok
direction = "forward"; // Fehler
`}/>

### Typinferenz

* TypeScript unterstützt Typinferenz
    * Wenn kein Typ angegeben ist, versucht TypeScript, den Typ automatisch zu erkennen
    * Beispiel:

<LiveJavascript
    language="typescript"
    enableTypeDiagnostics={true}
    initialCode={`
let message = "Hello, World!"; // TypeScript erkennt den Typ als string
let count = 42; // TypeScript erkennt den Typ als number
let isActive = true; // TypeScript erkennt den Typ als boolean
`} />

* Typinferenz funktioniert auch bei Funktionen

<LiveJavascript
    language="typescript"
    enableTypeDiagnostics={true}
    initialCode={`
function multiply(a: number, b: number) {
    return a * b; // TypeScript erkennt den Rückgabetyp als number
}

function applyToSelf(a: number, operation: (a: number, b: number) => number) {
    return operation(a, a); // TypeScript erkennt den Rückgabetyp als number
}

// Nur weil TypeScript den Rückgabewert von multiply als number erkennt, kann die Funktion als Parameter übergeben werden
let result = applyToSelf(5, multiply); // result ist number
`} />

### Generics

* TypeScript unterstützt Generics
* Generics ermöglichen es, Funktionen und Klassen zu erstellen, die mit verschiedenen Typen arbeiten können, ohne den Typ im Voraus festlegen zu müssen

* Beispiel ohne Generics:

<LiveJavascript
    language="typescript"
    enableTypeDiagnostics={true}
    initialCode={`
function firstNumber(items: number[]): number {
  return items[0];
}

function firstString(items: string[]): string {
  return items[0];
}

const n = firstNumber([10, 20, 30]); // n = 10
const s = firstString(["a", "b", "c"]); // s = "a"
`}/>

* Beispiel mit Generics:

<LiveJavascript
    language="typescript"
    enableTypeDiagnostics={true}
    initialCode={`
function first<T>(items: T[]): T {
  return items[0];
}

const n = first([10, 20, 30]); // n = 10
const s = first(["a", "b", "c"]); // s = "a"
`}/>

* In der Praxis werden Generics häufig in Bibliotheken und Frameworks verwendet, um wiederverwendbaren und typensicheren Code zu schreiben
* TypeScript unterstützt auch fortgeschrittene Typen wie bedingte Typen, Mapped Types und Utility Types
* Diese werden jedoch in der Praxis seltener verwendet und sind eher für fortgeschrittene Anwendungsfälle gedacht
* Für die meisten Anwendungsfälle reichen die oben genannten Typen und Konzepte aus

### Wie man sich selbst ins Knie schießen kann

* TypeScript unterstützt die "Typisierung" als `any` und das Type Casting mittels `as`
* `any` deaktiviert effektiv die Typprüfung für die Variable


<LiveJavascript
    language="typescript"
    enableTypeDiagnostics={true}
    initialCode={`
function add(a: number, b: number): number {
  return a + b;
}

const someString = "Hello, World!";
const actuallyNotANumber = someString as any as number;

console.log(add(1, actuallyNotANumber)); // KEIN TS Fehler, aber auch nicht schlau
`}
/>

* `as` zwingt den Compiler, eine Variable als einen bestimmten Typ zu behandeln
* Beide sollten mit Vorsicht verwendet werden, da sie die Vorteile von TypeScript untergraben können
* In der Praxis bin ich immer sehr vorsichtig, wenn ich `any` oder `as` sehe
* In der Regel ist die Nutzung davon ein Indikator dafür, dass entweder das Verständnis oder die Motivation für sauberes Typisieren fehlte

## Setup eines TypeScript Projekts

* Voraussetzung: Node.js und npm sind installiert
* Das Terminal befindet sich in einem leeren Verzeichnis
* Initialisiere ein neues Node.js Projekt:

```bash
npm init -y
npm install --save-dev typescript @types/node
npx tsc --init
```

* Dabei wird eine `tsconfig.json` erstellt
* In der Regel kann die `tsconfig.json` belassen werden, wie sie ist und nur angepasst werden, was für das Projekt notwendig ist
* Ein paar wichtige Optionen sollten jedoch erklärt werden

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "node",
    "rootDir": "src",
    "outDir": "dist",
    "strict": true,
    "esModuleInterop": true
  }
}
```

* Optionen erklärt:
    * `target`: JavaScript Version, die generiert wird
        * Hier: ES2022 (aktuelle Version mit modernen Features)
    * `module`: Modul-System (ESNext für moderne Module)
        * Hier: ESNext (aktuelle Version mit modernen Features)
        * `CommonJS`: Klassisches Node.js Modul-System
            * Wird in älteren Projekten oder bei Nutzung von älteren Bibliotheken verwendet
            * Konkrete Unterschiede:
                * `import fs from 'fs'` (ESNext) vs. `const fs = require('fs')` (CommonJS)
                * `export function foo() {}` (ESNext) vs. `module.exports = { foo }` (CommonJS)
    * `moduleResolution`: Wie Module aufgelöst werden (Node.js Stil)
        * Hier: node
        * Konkrete Auswirkung:
            * `import { something } from 'some-package'` sucht in `node_modules/some-package`
        * Standardwert ist `classic`, was in Browser-Umgebungen genutzt wird
            * Sucht nicht in `node_modules`, sondern nur relative Pfade
    * `rootDir`: Quellcode-Verzeichnis
    * `outDir`: Ausgabe-Verzeichnis für transpilierten Code
    * `strict`: Aktiviert strenge Typprüfungen
        * Empfohlen für bessere Codequalität
        * Konkrete Auswirkungen (einige Beispiele):
            * `noImplicitAny`: Verhindert implizite `any` Typen
            * `strictNullChecks`: Verhindert die Zuweisung von `null` oder `undefined` zu anderen Typen
            * `strictFunctionTypes`: Strengere Überprüfung von Funktionsparametern
            * `strictPropertyInitialization`: Stellt sicher, dass Klassen-Eigenschaften im Konstruktor initialisiert werden
            * `noImplicitThis`: Verhindert die Verwendung von `this` mit dem Typ `any`
    * `esModuleInterop`: Ermöglicht den Import von CommonJS Modulen
        * Konkrete Auswirkungen:
            * Ermöglicht `import fs from 'fs'` anstelle von `import * as fs from 'fs'`

* Auszug aus einer passenden `package.json`

```json
{
  "name": "my-typescript-project",
  "version": "1.0.0",
  "main": "dist/index.js",
  "type": "module",
  "scripts": {
    "build": "tsc",
    "start": "node dist/index.js",
    "dev": "tsc --watch"
  },
  "devDependencies": {
    "@types/body-parser": "^1.19.6",
    "@types/express": "^5.0.3",
    "@types/node": "^24.6.2",
    "typescript": "^5.9.3"
  },
  "dependencies": {
    "body-parser": "^2.2.0",
    "express": "^5.1.0"
  }
}
```

`src/index.ts`:

```typescript
import express from 'express';

const app = express();
const PORT = 8000;

app.get('/', (req, res) => {
    res.json({ data: 'Hello World!' });
});
app.listen(PORT, () => {
    console.log(`Server is running on http://localhost:${PORT}`);
});
```

* Der Code ist auf dem ersten Blick identisch zu dem, was wir ohne Typescript genutzt haben
* Der einzige Unterschied ist, dass die Datei jetzt `index.ts` heißt
* Aber: TypeScript wird den Code auf Typfehler überprüfen
* Express ist in TypeScript geschrieben und stellt Typdefinitionen bereit
* Daher sind `req` und `res` automatisch typisiert und IDEs sowie der TypeScript Compiler wissen, welche Methoden und Eigenschaften verfügbar sind
* klarer wird es, wenn wir versuchen, etwas Falsches zu tun:

```typescript
app.get('/', (req, res) => {
    res.json({ data: 'Hello World!' });
    res.notAMethod('This will cause an error'); // Fehler: TS2339: Property notAMethod does not exist on type Response<any, Record<string, any>, number>
});
```
