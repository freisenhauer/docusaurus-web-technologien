## 1. Was ist das "Web"? - Prinzipien und Vision

### Die ursprüngliche Vision

* **Tim Berners-Lee**, CERN, 1989: Wissenschaftler wollten Forschungsergebnisse elektronisch austauschen und miteinander verknüpfen.
* Kernidee: Ein **offenes, dezentrales Netzwerk** verlinkter Dokumente.
* Wichtig: **Jeder** kann teilnehmen, **jeder** kann publizieren - keine zentrale Kontrollinstanz.
* Das **World Wide Web** wurde geboren - ein System, das auf dem Internet aufsetzt, aber nicht mit ihm gleichzusetzen ist.
    * Das **Internet** ist die technische Infrastruktur (Netzwerke, Protokolle wie TCP/IP).
    * Das **Web** ist ein Dienst, der auf dem Internet läuft (HTTP, HTML, URLs).

### Die Prinzipien des Webs

Aus dieser Vision entstanden **fundamentale Prinzipien**, die das Web bis heute prägen:

* **Offenheit & Standards**:
    * Keine proprietären Technologien, keine Lizenzgebühren.
    * Offene Standards werden von Organisationen wie **W3C** (World Wide Web Consortium), **WHATWG** (Web Hypertext Application Technology Working Group) und **IETF** (Internet Engineering Task Force) entwickelt.
    * Jeder kann diese Standards implementieren - deshalb gibt es viele verschiedene Browser, die alle dieselben Webseiten anzeigen können.
    * Im Gegensatz zu geschlossenen Plattformen (App Stores, proprietäre Software) ist das Web für alle zugänglich.

* **Universeller Zugang**:
    * "Write once, run anywhere" - eine Webseite funktioniert auf **jedem Gerät** mit einem Browser.
    * Plattformunabhängig: Windows, macOS, Linux, Android, iOS, etc.
    * Der **Browser** als universelle Laufzeitumgebung macht dies möglich.
    * Keine Installation notwendig - einfach URL eingeben und loslegen.

* **Verlinkte Information**:
    * Das "Web" im "World Wide Web" - Inhalte sind durch **Hyperlinks** miteinander verbunden.
    * **URLs** (Uniform Resource Locators) als universelle Adressen für Ressourcen im Web.
    * Jede Information kann von überall verlinkt und referenziert werden.

* **Client-Server-Architektur**:
    * Ressourcen liegen auf **Servern** und werden auf Anfrage an **Clients** (Browser) ausgeliefert.
    * Verteilte Systeme von Anfang an - nicht alles muss lokal installiert sein.
    * Der Server stellt Ressourcen bereit, der Client (Browser) fordert sie an und stellt sie dar.

## 2. Von Dokumenten zu Applikationen - Eine kurze Evolution

Das Web hat sich seit seiner Entstehung dramatisch weiterentwickelt.
Diese Evolution zeigt, wie aus einem System zum Austausch von Dokumenten eine universelle Plattform für Software wurde.

### Phase 1: Statische Dokumente (frühe 1990er)

* **HTML** für Textformatierung und einfache Strukturen.
* Webseiten waren im Wesentlichen digitale Dokumente - Text, Bilder, Links.
* Einfach, aber stark limitiert: Keine Interaktion, keine Dynamik.
* Beispiele: Wissenschaftliche Publikationen, einfache Informationsseiten.

### Phase 2: Interaktivität (Mitte/Ende 1990er)

* **JavaScript**, Formulare, dynamische Inhalte ermöglichen erste Interaktionen.
* Nutzer können Daten eingeben, Formulare absenden, einfache Berechnungen durchführen.
* **Problem entsteht**: Mit Interaktivität kommen neue **Sicherheitsrisiken**.
    * Wenn Nutzer Daten eingeben können, können Angreifer schädliche Daten eingeben.
    * Erste Cross-Site-Scripting (XSS) und SQL-Injection Angriffe tauchen auf.
    * → **Vorschau**: Deshalb widmen wir Kapitel 5 der Sicherheit!

### Phase 3: Web-Applikationen (2000er)

* **Gmail** (2004), **Google Maps** (2005): Der Browser wird zur Plattform für komplexe Anwendungen.
* **AJAX** (Asynchronous JavaScript and XML) ermöglicht das Nachladen von Daten ohne Seitenrefresh.
* **Single Page Applications (SPAs)**: Die Webseite verhält sich wie eine Desktop-Anwendung.
* **Sicherheitsproblem verschärft sich**:
    * Sensible Daten (E-Mails, Standorte) im Browser.
    * Authentifizierung wird kritisch - wer darf was sehen?
    * Zahlungen, persönliche Informationen im Web.
    * → **Sicherheit wird zum zentralen Thema** (Kapitel 5).
* Gleichzeitig wird das Web und damit einhergehende **Herausforderungen immer komplexer**:
    * Verschiedene Browser mit unterschiedlichen Implementierungen.
    * Performance-Optimierung wird wichtig (Ladezeiten, Reaktionsfähigkeit).
    * Skalierbarkeit der Server-Infrastruktur.

### Phase 4: Das Web als universelle Plattform (heute)

* **Progressive Web Apps (PWAs)**: Web-Apps, die sich fast wie native Apps verhalten.
    * Offline-Funktionalität, Push-Benachrichtigungen, Installation auf dem Homescreen.
    * Zugriff auf Hardware: Kamera, Mikrofon, Standort, Bluetooth, USB.
* **WebAssembly**: Hochperformanter Code im Browser (C++, Rust).
* **Web-APIs**: WebGL (3D-Grafik), WebRTC (Video-Chat), WebGPU (GPU-Zugriff).
* **Sicherheit wird kritisch**:
    * Banking-Apps, Gesundheitsdaten, E-Commerce - alles im Browser.
    * Milliarden von Nutzern weltweit - ein attraktives Ziel für Angreifer.
    * → **Deshalb ist Kapitel 5 (Sicherheit) so wichtig**.

## 3. Warum Web-Technologien lernen? - Die Unvermeidbarkeit des Webs

### Das Web ist allgegenwärtig

* Über **5,5 Milliarden Menschen** nutzen das Internet und das Web täglich (Stand 2024).
* Fast jede moderne Software hat eine **Web-Komponente**:
    * Mobile Apps kommunizieren mit Web-Services.
    * Desktop-Anwendungen nutzen Web-APIs für Authentifizierung, Updates, Daten.
    * IoT-Geräte (Smart Home, Wearables) haben oft Web-Interfaces.
* Das Web ist die **Infrastruktur** der modernen digitalen Welt.

### Web-Technologien sind die universellste Plattform

**Eine Skillset → viele Einsatzgebiete:**

* **Klassische Websites**: Unternehmenswebseiten, Blogs, Portfolios.
* **Web-Applikationen**: Gmail, Figma, Notion, VS Code Web, Google Docs.
* **Mobile Apps**:
    * Progressive Web Apps (PWAs).
    * Hybrid-Apps mit React Native, Ionic, Capacitor.
* **Desktop-Applikationen**:
    * Electron-Apps: Slack, Discord, VS Code, Spotify.
* **Backend-Services**:
    * Node.js, Deno, Bun - JavaScript/TypeScript auf dem Server.
* **IoT-Interfaces**:
    * Konfiguration von Routern, Smart-Home-Geräten über Web-Interfaces.

**Keine andere Technologie ist so vielseitig** - mit Web-Technologien kannst du Software für praktisch jede Plattform entwickeln.

### Der Arbeitsmarkt

* **Web-Entwickler** gehören zu den **gefragtesten Berufen** in der IT-Branche.
* Jedes Unternehmen braucht eine **Web-Präsenz** - sei es eine einfache Website oder eine komplexe Applikation.
* **Startups** beginnen meistens mit einer Web-App:
    * Schnell zu entwickeln.
    * Günstig (keine plattformspezifische Entwicklung).
    * Plattformübergreifend (Desktop, Mobile).
* Laut Statistiken gehören **Frontend-**, **Backend-** und **Full-Stack-Entwickler** zu den **Top 10 der gefragtesten Tech-Jobs**.

### Offenheit = Zugänglichkeit

* **Keine teuren Lizenzen** erforderlich (vs. proprietäre Plattformen wie iOS-Entwicklung).
* **Riesige Community**: Millionen von Entwicklern, unzählige Open-Source-Projekte.
* **Kostenlose Ressourcen**: Tutorials, Dokumentationen, Frameworks, Libraries.
* **Niedrige Einstiegshürde**:
    * Browser + Texteditor = alles, was du zum Starten brauchst.
    * Kein teures Setup, keine spezielle Hardware.

### Web-Technologien entwickeln sich rasant

* Was früher nur **native Apps** konnten, geht jetzt im **Browser**:
    * **3D-Grafik**: WebGL, Three.js - komplexe Spiele und Visualisierungen.
    * **Video-Bearbeitung**: Adobe Premiere Rush, Clipchamp - komplett im Browser.
    * **Design-Tools**: Figma, Canva - professionelle Design-Software ohne Installation.
    * **Hardware-Zugriff**: WebUSB, WebBluetooth, WebGPU - direkte Hardware-Kommunikation.
* Das Web wird nicht verschwinden, sondern **mächtiger** und **unverzichtbarer**.

### Der entscheidende Punkt: Selbst native Apps brauchen das Web

* Selbst wenn du später **native Apps** entwickelst (iOS, Android, Desktop), kommst du am Web nicht vorbei:
    * **Backend-APIs** basieren auf Web-Technologien: HTTP, REST, GraphQL.
    * **Authentifizierung** läuft meist über Web-Standards: OAuth, JWT, OpenID Connect.
    * **Cloud-Infrastruktur** nutzt Web-Protokolle: AWS, Azure, Google Cloud.
    * **Updates, Analytics, Monitoring** - alles über Web-Services.

**Web-Technologien sind das Fundament der modernen Software-Entwicklung** - du kannst ihnen nicht entkommen, also solltest du sie beherrschen.

## 4. Web vs. Native - Warum Web-Technologien?

Um die **Besonderheiten** und **Vorteile** von Web-Technologien zu verstehen, hilft ein Vergleich mit nativen Applikationen.

### Native Applikationen

* **Definition**: Software, die direkt auf dem Betriebssystem läuft.
* **Beispiele**:
    * Windows: `.exe`-Dateien.
    * macOS: `.app`-Bundles.
    * Android: `.apk`-Dateien.
    * iOS: Apps aus dem App Store.

**Vorteile:**
* **Voller Hardware-Zugriff**: Direkter Zugriff auf Sensoren, Bluetooth, Dateisystem, Kamera, etc.
* **Hohe Performance**: Näher an der Hardware, optimiert für das spezifische Betriebssystem.
* **Offline-Funktionalität**: Funktioniert ohne Internetverbindung (bei entsprechendem Design).

**Nachteile:**
* **Plattformabhängig**:
    * Windows-App läuft nicht auf macOS.
    * Android-App läuft nicht auf iOS.
    * → Jede Plattform braucht eigenen Code, eigenes Team, eigene Wartung.
* **Installation & Updates kompliziert**:
    * Nutzer müssen die App herunterladen und installieren.
    * Updates müssen manuell installiert werden (oder komplex automatisiert).
* **Gatekeeper**:
    * App Stores (Apple, Google) kontrollieren, was veröffentlicht werden darf.
    * Gebühren (z.B. 30% bei Apple), Review-Prozesse, Zensur möglich.

### Web-Applikationen

* **Definition**: Software, die im Browser läuft und über das Web ausgeliefert wird.
* **Beispiele**: Gmail, Google Maps, Figma, Notion, YouTube, Twitter/X.

**Vorteile:**
* **Plattformunabhängig**:
    * Eine Codebasis → funktioniert auf Windows, macOS, Linux, Android, iOS, etc.
    * → Ein Team, ein Code, alle Plattformen.
* **Keine Installation notwendig**:
    * URL eingeben, loslegen.
    * Nutzer hat immer die **neueste Version** (Server aktualisiert).
* **Offen**:
    * Kein Gatekeeper, keine Review-Prozesse (außer bei PWA-Installation in App Stores).
    * Jeder kann eine Webseite veröffentlichen.
* **Schnelle Iteration**:
    * Bug gefunden? Fix deployen, alle Nutzer haben ihn sofort.
    * Kein Warten auf App-Store-Approval.

**Nachteile:**
* **Eingeschränkter Hardware-Zugriff** (wird besser):
    * Traditionell: Kein Zugriff auf Dateisystem, Bluetooth, USB.
    * Heute: WebUSB, WebBluetooth, File System Access API - vieles wird möglich.
* **Abhängig von Internetverbindung**:
    * Traditionell: Ohne Internet nichts los.
    * Heute: PWAs mit Service Workers ermöglichen Offline-Funktionalität.
* **Performance-Einschränkungen**:
    * Browser als Zwischenschicht kann Performance kosten.
    * Aber: WebAssembly bringt Near-Native-Performance.

### Hybrid-Ansätze: Das Beste aus beiden Welten

* **Electron**:
    * Baut Web-Technologien (HTML, CSS, JavaScript) zu Desktop-Apps.
    * Beispiele: VS Code, Slack, Discord, Spotify.
* **React Native, Ionic, Capacitor**:
    * Baut Web-Technologien zu nativen Mobile Apps.
* **Progressive Web Apps (PWAs)**:
    * Web-Apps, die wie native Apps funktionieren (Offline, Push-Benachrichtigungen, Installation).

**Das zeigt**: Web-Technologien sind so universell, dass sie sogar für native Apps verwendet werden.
