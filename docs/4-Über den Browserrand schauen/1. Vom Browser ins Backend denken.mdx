## 1. Warum reicht der Browser nicht aus?

Bisher haben wir uns vor allem mit dem Frontend beschäftigt - HTML, CSS und JavaScript im Browser. Aber was, wenn du Daten speichern willst? Was, wenn Nutzer miteinander interagieren sollen? Was, wenn Inhalte personalisiert werden müssen?

**Die Grenzen des Browsers:**
* **Keine dauerhafte Speicherung**: LocalStorage und Cookies sind auf den einzelnen Browser beschränkt und können vom Nutzer gelöscht werden
* **Keine zentrale Datenverwaltung**: Jeder Browser hat seine eigenen Daten, aber wie teilst du Daten zwischen verschiedenen Nutzern?
* **Keine sensiblen Operationen**: Passwort-Hashing, Zahlungsabwicklung oder API-Keys sollten niemals im Browser liegen
* **Keine Authentifizierung**: Wer ist der Nutzer wirklich? Der Browser kann das nicht zuverlässig prüfen

Genau hier kommt das Backend ins Spiel - ein Server, der zentral Daten verwaltet, Geschäftslogik ausführt und als Single Source of Truth fungiert.

:::info
Der Browser ist nur eine Schnittstelle zum Nutzer. Die "Wahrheit" über Daten, Berechtigungen und Zustände liegt immer auf dem Server.
:::

## 2. Statische Inhalte vs. dynamische Inhalte

### Statische Inhalte

**Statische Inhalte** sind Dateien, die sich nicht verändern und direkt vom Server an den Browser ausgeliefert werden - genau so, wie sie auf dem Server gespeichert sind.

**Eigenschaften statischer Inhalte:**
* **Unveränderlich**: Dieselbe Datei wird an alle Nutzer ausgeliefert
* **Schnell**: Keine Verarbeitung nötig, direkter Download
* **Skalierbar**: Können problemlos auf Content Delivery Networks (CDNs) verteilt werden
* **Cachebar**: Browser und CDNs können sie effizient zwischenspeichern

**Beispiele:**
* HTML-Dateien (z.B. `index.html`)
* CSS-Dateien (z.B. `styles.css`)
* Bilder (z.B. `logo.png`, `hero.jpg`)
* PDF-Dokumente
* JavaScript-Dateien (die Datei selbst ist statisch, auch wenn der Code dynamisch ist)

**Request-Flow bei statischen Inhalten:**
```
Browser                    Webserver
   │                          │
   │──── GET /index.html ────>│
   │                          │
   │<───── index.html ────────│
   │      (200 OK)            │
```

**Limitierungen statischer Inhalte:**
* **Keine Personalisierung**: Jeder Nutzer sieht dasselbe
* **Keine Datenpersistenz**: Keine Möglichkeit, Nutzerdaten zentral zu speichern
* **Keine Nutzer-Interaktion**: Kein Chat, keine Kommentare, kein Social Network
* **Keine geschützte Logik**: Alles liegt offen im Browser

### Dynamische Inhalte

**Dynamische Inhalte** werden zur Laufzeit generiert und können sich je nach Nutzer, Kontext oder Daten unterscheiden.

:::tip
Denk an YouTube: Die Video-Seite ist für jeden Nutzer anders - abhängig von deinen Abos, deiner Watch-History und deinen Empfehlungen. Das ist dynamischer Content.
:::

**Vorteile dynamischer Inhalte:**
* **Personalisierung**: Jeder Nutzer bekommt seine individuellen Inhalte
* **Datenpersistenz**: Daten werden in Datenbanken gespeichert
* **Interaktivität**: Nutzer können miteinander kommunizieren
* **Authentifizierung & Autorisierung**: Geschützte Bereiche und Rollen-Management
* **Geschäftslogik**: Komplexe Berechnungen und Validierungen auf dem Server

## 3. Ansätze für dynamische Inhalte

Es gibt verschiedene Architektur-Ansätze, um dynamische Inhalte bereitzustellen:

### Server Side Rendering (SSR)

**Prinzip**: Der Server generiert für jede Anfrage eine vollständige HTML-Seite und sendet sie an den Browser.

**Request-Flow:**
```
Browser                    Webserver                 Datenbank
   │                          │                          │
   │─── GET /profile ────────>│                          │
   │                          │                          │
   │                          │──── Query User Data ────>│
   │                          │<──── User Data ──────────│
   │                          │                          │
   │                          │ (Generiere HTML)         │
   │<──── HTML Seite ─────────│                          │
   │      (200 OK)            │                          │
```

**Eigenschaften:**
* **Vollständige Seite**: Jede Navigation lädt eine neue HTML-Seite vom Server
* **SEO-freundlich**: Suchmaschinen können den fertigen HTML-Content crawlen
* **Schneller First Paint**: Die erste Seite wird schnell angezeigt
* **Server-Last**: Jede Anfrage erfordert Server-Verarbeitung

**Beispiele:**
* **PHP**: Das klassische Beispiel für SSR (z.B. WordPress)
* **Ruby on Rails**: Web-Framework mit SSR
* **Django**: Python-Framework für SSR

### Client Side Rendering (CSR)

**Prinzip**: Der Server liefert eine minimale HTML-Seite mit JavaScript aus. Das JavaScript lädt dann Daten über APIs und rendert die Seite im Browser.

**Request-Flow:**
```
Browser                    Webserver                    API Server
   │                          │                             │
   │─── GET /app ───────────>│                             │
   │<──── index.html ─────────│                             │
   │      (minimal)           │                             │
   │                          │                             │
   │─── GET /app.js ─────────>│                             │
   │<──── app.js ─────────────│                             │
   │                          │                             │
   │ (JavaScript führt aus)   │                             │
   │                          │                             │
   │─── GET /api/user ───────────────────────────────────>│
   │<──── JSON Daten ──────────────────────────────────────│
   │                          │                             │
   │ (Rendert Seite)          │                             │
```

**Eigenschaften:**
* **API-basiert**: Frontend kommuniziert mit Backend über REST/GraphQL APIs
* **SPA (Single Page Application)**: Keine vollständigen Seiten-Reloads mehr
* **Interaktivität**: Flüssige Nutzer-Erfahrung wie in nativen Apps
* **Datenformate**: Meist JSON oder XML für API-Kommunikation
* **Flexible Backends**: APIs können in verschiedenen Sprachen implementiert sein (Node.js, Python, Go, Java, etc.)

**Beispiele:**
* **React**: JavaScript-Library für CSR
* **Vue.js**: Progressive Framework
* **Angular**: Vollständiges Framework

### Hybride Ansätze

**Prinzip**: Kombination aus SSR und CSR - das Beste aus beiden Welten.

**Eigenschaften:**
* **Initial SSR**: Erste Seite wird serverseitig gerendert (schneller First Paint)
* **Anschließend CSR**: Navigation erfolgt dann clientseitig (flüssige UX)
* **Flexible Strategie**: Manche Seiten SSR, andere CSR

**Beispiele:**
* **Next.js**: React-Framework mit SSR/CSR/SSG (Static Site Generation)
* **Nuxt.js**: Vue.js-Framework mit hybriden Ansätzen
* **SvelteKit**: Svelte-Framework mit flexiblem Rendering

### Vergleich: SSR vs. CSR vs. Hybrid

| Kriterium | Server Side Rendering (SSR) | Client Side Rendering (CSR) | Hybrid |
|-----------|----------------------------|----------------------------|---------|
| **First Paint** | Sehr schnell ⚡ | Langsamer | Sehr schnell ⚡ |
| **SEO** | Hervorragend ✅ | Schwierig ❌ | Hervorragend ✅ |
| **Interaktivität** | Bei jeder Navigation neu laden | Sehr flüssig ✅ | Sehr flüssig ✅ |
| **Server-Last** | Hoch bei jedem Request | Niedrig (nur API-Calls) | Mittel |
| **Komplexität** | Mittel | Mittel | Hoch |
| **Beispiel Use Case** | Content-Seiten, Blogs | Web-Apps, Dashboards | E-Commerce, Social Media |

## 4. Fokus dieser Vorlesung: API-basierte Architekturen

:::info Was du in dieser Vorlesung lernst
In dieser Vorlesung konzentrieren wir uns auf **API-basierte Architekturen** (CSR-Ansatz):
* **Backend**: Node.js mit TypeScript für REST-APIs
* **Frontend**: Kommuniziert mit dem Backend über HTTP-Requests
* **Datenformat**: JSON für die Datenübertragung
* **Vorteil**: Du lernst JavaScript/TypeScript sowohl im Frontend als auch im Backend - eine Sprache für alles!
:::

**Warum API-basiert?**
* **Separation of Concerns**: Frontend und Backend sind klar getrennt
* **Flexibilität**: Mehrere Frontends (Web, Mobile App) können dieselbe API nutzen
* **Moderne Architektur**: So arbeiten die meisten modernen Web-Apps (Instagram, Twitter, Netflix, etc.)
* **Skalierbarkeit**: Frontend und Backend können unabhängig skaliert werden
* **Didaktisch sinnvoll**: Ich kann auf viele Konzepte eingehen, ohne mich in Framework-spezifischen Details zu verlieren

**Architektur-Übersicht:**
```
┌─────────────────────────────────────────────────────────────┐
│                         Browser                             │
│                                                             │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  React/Vue/Vanilla JS (Frontend)                     │   │
│  │                                                      │   │
│  │  - Rendert UI                                        │   │
│  │  - Macht API-Calls (fetch/axios)                     │   │
│  │  - Verarbeitet JSON-Antworten                        │   │
│  └──────────────────────────────────────────────────────┘   │
│                          │                                  │
└──────────────────────────┼──────────────────────────────────┘
                           │ HTTP Request (JSON)
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                     Backend Server                          │
│                                                             │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  Node.js + TypeScript (Backend)                      │   │
│  │                                                      │   │
│  │  - REST API Endpoints                                │   │
│  │  - Geschäftslogik                                    │   │
│  │  - Authentifizierung                                 │   │
│  │  - Datenbankzugriff                                  │   │
│  └──────────────────────────────────────────────────────┘   │
│                          │                                  │
└──────────────────────────┼──────────────────────────────────┘
                           │
                           ▼
                   ┌───────────────┐
                   │   Datenbank   │
                   │  (PostgreSQL) │
                   └───────────────┘
```

In den nächsten Kapiteln lernst du, wie du ein solches Backend mit Node.js und TypeScript aufbaust, APIs entwickelst, Datenbanken anbindest und deine Anwendung absicherst.
