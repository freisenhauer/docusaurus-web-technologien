Bisher haben wir JavaScript im Browser kennengelernt.
JavaScript kann jedoch auch für Backend-Entwicklung genutzt werden.
Hier kommt **Node.js** ins Spiel – eine Laufzeitumgebung, die JavaScript außerhalb des Browsers ermöglicht.

## Was ist Node.js?

* **JavaScript außerhalb des Browsers**: Node.js ist eine JavaScript-Laufzeitumgebung, die es ermöglicht, JavaScript-Code außerhalb des Browsers auszuführen.
* **V8 Engine**: Node.js basiert auf der V8 JavaScript-Engine von Google Chrome, die JavaScript in Maschinencode kompiliert und somit eine hohe Ausführungsgeschwindigkeit ermöglicht.
* **Serverseitige Module**: Ergänzend zu den JavaScript-Sprachfeatures bietet Node.js eine Vielzahl von Modulen und Bibliotheken, die speziell für serverseitige Anwendungen entwickelt wurden.
* **Ziel**: JavaScript für Server, Skripte, CLI-Tools, APIs, etc nutzbar machen
* **Vorteile**:
    * **Einheitliche Sprache**: JavaScript für Frontend und Backend nutzbar
    * **Non-blocking I/O**: Asynchrones I/O Modell, gut für APIs, die viele Anfragen gleichzeitig bearbeiten müssen
    * **Große Community**: Viele Pakete und Bibliotheken über npm verfügbar
    * **Schneller Einstieg**: Sehr leichtgewichtig, schnell erste Ergebnisse
* **Nachteile**:
    * **Single Threaded**: Nicht ideal für CPU-intensive Aufgaben, aber in Microservice Architekturen weniger schlimm

## Architekturprinzip: Event Loop & non-blocking I/O

### Traditionell vs. Node.js

* **Klassischer Ansatz**: Pro Request ein Thread/Prozess; hoher Speicher-Overhead und Context-Switching
* **Node.js Ansatz**: Ein Thread mit Event Loop; Tasks in Queue, I/O Operationen werden asynchron ausgeführt

### Wie funktioniert die Event Loop?

Die **Event Loop** ist das Herzstück von Node.js. Sie überwacht eine Queue von Tasks und führt sie aus, sobald sie bereit sind:

```
   ┌───────────────────────────┐
┌─>│           timers          │  setTimeout(), setInterval()
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │     pending callbacks     │  I/O Callbacks (z.B. Netzwerk)
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │       idle, prepare       │  Interne Operationen
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │           poll            │  Neue I/O Events abrufen
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │           check           │  setImmediate() Callbacks
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
└──┤      close callbacks      │  socket.on('close', ...)
   └───────────────────────────┘
```

**Event Loop Phasen**:

* **Timers**: Ausführung von Callbacks, die durch `setTimeout()` oder `setInterval()` geplant wurden
* **I/O Callbacks**: Ausführung von I/O-bezogenen Callbacks (z.B. Netzwerkoperationen)
* **Idle, Prepare**: Interne Operationen von Node.js
* **Poll**: Abrufen neuer I/O-Ereignisse; Ausführung von I/O-bezogenen Callbacks
* **Check**: Ausführung von Callbacks, die durch `setImmediate()` geplant wurden
* **Close Callbacks**: Ausführung von Close-Event-Callbacks (z.B. `socket.on('close', ...)`)

### Konsequenzen für die Entwicklung

:::danger Blockierende Operationen vermeiden!
* **Asynchrone I/O**: Lese/Schreib-Operationen (DB, File, Netzwerk) werden asynchron über Callbacks/Promises/async-await gehandhabt
* **Blockierende Operationen**: Blockierende Operationen (z.B. große JSON-Dateien synchron lesen) bremsen den gesamten Event Loop aus und **sind zwingend zu vermeiden**
* **Single-Threaded**: Da Node.js single-threaded ist, blockiert synchroner Code ALLE anderen Anfragen
:::


## Einstieg: ein Minimalprojekt mit npm

:::info
In der [npm Dokumentation](https://docs.npmjs.com/downloading-and-installing-node-js-and-npm) sind Anleitungen für jedes Betriebssystem zu finden, wie `npm` installiert werden kann.
Ich empfehle die Installation eines Node Version Managers (nvm), um verschiedene Node.js Versionen parallel nutzen zu können.

**Ein Node.js Projekt in einem eigenen Verzeichnis anlegen:**

```bash
# Neues Verzeichnis erstellen und hineinwechseln
mkdir myapp && cd myapp

# package.json initialisieren (mit Standardwerten)
npm init -y

# Express.js Framework installieren
npm install express
```

**Die `package.json` Datei erklärt:**

```json
{
  "name": "myapp",
  "version": "1.0.0",
  "main": "index.js",
  "scripts": {
    "start": "node src/index.js"
  },
  "dependencies": {
    "express": "^5.1.0"
  }
}
```

* **`name`**: Name des Projekts
* **`version`**: Versionsnummer des Projekts (Semantic Versioning)
* **`main`**: Einstiegspunkt der Anwendung
* **`scripts`**: Kommandos, die über `npm run` ausgeführt werden können
* **`dependencies`**: Produktions-Abhängigkeiten des Projekts

### Ein erster HTTP Webserver (nativ)

`index.js`:

```js
import http from 'http';

// HTTP-Server erstellen
const server = http.createServer((req, res) => {
    // Response-Header setzen
    res.writeHead(200, { 'Content-Type': 'application/json' });

    // JSON-Response senden
    res.end(JSON.stringify({
        data: 'Hello World!'
    }));
});

// Server auf Port 8000 starten
server.listen(8000);
```

### Webserver mit Express.js (empfohlen)

`src/index.js`:

```js
import express from 'express';

const app = express();
const PORT = 8000;

// Route für GET-Anfragen auf "/"
app.get('/', (req, res) => {
    res.json({ data: 'Hello World!' });
});

// Server starten
app.listen(PORT, () => {
    console.log(`Server läuft auf http://localhost:${PORT}`);
});
```

## Tooling & Dev-Komfort

**`nodemon` für automatischen Neustart bei Dateiänderungen:**

Bei jeder Code-Änderung den Server manuell neu zu starten ist mühsam. `nodemon` überwacht Dateien und startet den Server automatisch neu:

```bash
# nodemon als Development-Dependency installieren
npm install --save-dev nodemon
```

**`package.json` erweitern:**

```json
{
    "scripts": {
        "start": "node src/index.js",
        "dev": "nodemon src/index.js"
    }
}
```

Jetzt kannst du den Entwicklungsserver mit `npm run dev` starten. Bei jeder Änderung startet `nodemon` die Anwendung automatisch neu.

## Zusammenfassung

* **Node.js**: JavaScript-Laufzeitumgebung außerhalb des Browsers, basierend auf der V8 Engine
* **Event Loop**: Single-threaded, non-blocking I/O Architektur ermöglicht hohe Skalierbarkeit für I/O-intensive Anwendungen
* **Asynchronität ist Pflicht**: Blockierende Operationen müssen zwingend vermieden werden, da sie den gesamten Event Loop blockieren
* **npm**: Paketmanager für Node.js mit riesigem Ökosystem an Bibliotheken
* **Express.js**: Minimalistisches, flexibles Web-Framework für Node.js
* **Development Tools**: `nodemon` für automatisches Neuladen während der Entwicklung
