Das moderne Web basiert auf einer Vielzahl von Protokollen und Technologien, die zusammenarbeiten, um Daten zwischen Clients und Servern auszutauschen. In diesem Kapitel werfen wir einen Blick auf die wichtigsten Grundbausteine des Internets – von den Schichten des TCP/IP-Modells über HTTP und REST bis hin zu den Details von URLs und MIME Types. Du wirst verstehen, wie Daten strukturiert, adressiert und übertragen werden, und warum diese Konzepte für die Entwicklung moderner Web-Applikationen fundamental sind.

## 1. Kurze Vorschau auf das Transmission Control Protocol/Internet Protocol (TCP/IP) Modell

* Das **TCP/IP Modell** ist ein Protokollmodell, das die Kommunikation im Internet beschreibt.
* **Vier Schichten**: Es besteht aus vier Schichten, die jeweils spezifische Aufgaben in der Datenübertragung übernehmen.
* **Hierarchischer Aufbau**: Die einzelnen Schichten bauen aufeinander auf
* **Tiefergehende Anforderungen**: Jede Schicht erfüllt tiefergehende Anforderungen an die Kommunikation als die darunterliegende Schicht.
* **Unabhängigkeit**: Dieser Aufbau ermöglicht bspw. Übertragungsprotokolle unabhängig von den darunterliegenden Technologien zu entwickeln. Bspw. ist dies der Grund, warum die gleichen höher liegenden Protokolle sowohl für Ethernet als auch für WLAN verwendet werden können.
* **DARPA**: Das Modell wurde von der DARPA (Defense Advanced Research Projects Agency) entwickelt und stellt derzeit das Standardmodell für die Kommunikation im Internet dar.
* **Alternative**: Alternativ existiert noch das ISO OSI Modell, welches jedoch in der Praxis kaum noch verwendet wird.
* **Vertiefung**: Weitere Behandlung findet im Modul "Rechnernetze" statt

### Die 4 Schichten des (TCP/IP) Modells

```
┌─────────────────────────────────────────┐
│   Schicht 4: Anwendungsschicht          │
│   (HTTP, FTP, SMTP, DNS)                │
├─────────────────────────────────────────┤
│   Schicht 3: Transportschicht           │
│   (TCP, UDP)                            │
├─────────────────────────────────────────┤
│   Schicht 2: Internetschicht            │
│   (IP, ICMP)                            │
├─────────────────────────────────────────┤
│   Schicht 1: Netzwerkzugriffsschicht    │
│   (Ethernet, WLAN)                      │
└─────────────────────────────────────────┘
```

* **Schicht 1: Netzwerkzugriffsschicht**
    * **Physikalische Übertragung**: Beinhaltet die physikalische Übertragung von Bits und die Sicherung der fehlerfreien Übertragung
    * Beantwortet die Fragen:
        * Was muss mit dem Kabel oder der Luft passieren, um 0 und 1 zu übertragen?
        * Was muss passieren, um sicherzustellen, dass die 0 und 1 korrekt ankommen?
    * Beispiele: Ethernet, WLAN

* **Schicht 2: Internetschicht**
    * **Routing**: Verantwortlich für das Routing von Datenpaketen zwischen Netzwerken
    * Beantwortet die Frage:
        * Wie finden die Daten den Weg von einem Gerät zum richtigen Empfänger - auch über viele Zwischenstationen?
    * Beispiele: IP (Internet Protocol), ICMP (Internet Control Message Protocol)

* **Schicht 3: Transportschicht**
    * **Zuverlässige Übertragung**: Gewährleistung der zuverlässigen Übertragung von Datenrahmen
    * Gewährleistet die zuverlässige Übertragung von Datenströmen
    * **Session-Management**: Verwaltung von Sitzungen zwischen Anwendungen
    * Beantwortet die Fragen:
        * Wie stellen wir sicher, dass alle Datenpakete vollständig, richtig und in der richtigen Reihenfolge ankommen?
        * Wie behalten wir den Überblick, wer mit wem spricht, welche Datenpakete zu welchem Gespräch gehören und wo wir im Gespräch gerade sind?
    * Beispiele: TCP, UDP

* **Schicht 4: Anwendungsschicht**
    * **Datenformatierung**: Formatierung und Übersetzung von Daten für die Anwendung
    * **Dienste**: Bereitstellung von Diensten für Anwendungen
    * Beantwortet die Fragen:
        * Wie müssen die Daten aufbereitet sein, damit Sender und Empfänger sie verstehen können?
        * Welche Information/Anfrage möchte die Anwendung senden und welche Antwort erwartet sie?
    * Beispiele: HTTP, FTP, SMTP, DNS

### Relevant für Web Technologien

* Besonders relevante Protokolle für Web Technologien sind:
    * **IP** (Internet Protocol) - Vermittlungsschicht
    * **TCP** (Transmission Control Protocol) - Transportschicht
    * **HTTP** (Hypertext Transfer Protocol) - Anwendungsschicht
    * **HTTPS** (HTTP Secure) - Anwendungsschicht (sichere Variante von HTTP)
    * **DNS** (Domain Name System) - Anwendungsschicht (Übersetzung von Domainnamen in IP-Adressen)

### Internet Protocol (IP)

* **IP** ist das grundlegende Protokoll für die Adressierung und das Routing von Datenpaketen im Internet.
* **Netzwerkübergreifend**: Es ermöglicht die Kommunikation zwischen Geräten über verschiedene Netzwerke hinweg.
* **IP-Adressen** sind eindeutige Identifikatoren für Geräte im Internet.
* **Erreichbarkeit**: Jede Webseite, jede Web Applikation und jeder Dienst im Internet oder Intranet ist über eine IP-Adresse erreichbar.
* **IP-Pakete** enthalten die Quell- und Ziel-IP-Adressen sowie die Nutzdaten.
* **Zwei Versionen**: Dabei muss unterschieden werden zwischen IPv4 (32-Bit Adressen) und IPv6 (128-Bit Adressen).
* **IPv4** ist das aktuell noch am weitesten verbreitete Protokoll
* Aufgrund der begrenzten Anzahl an Adressen wird zunehmend auf **IPv6** umgestellt.

```
  IPv4 (32-Bit)                    IPv6 (128-Bit)
  ┌────────────────────┐          ┌──────────────────────────────────┐
  │  192.168.1.1       │          │  2001:0db8:85a3:0000:0000:8a2e:  │
  │                    │          │  0370:7334                       │
  │  4,3 Milliarden    │    →     │  340 Sextillionen (36 Nullen)    │
  │  Adressen          │          │  Adressen                        │
  │  (erschöpft)       │          │  (praktisch unbegrenzt)          │
  └────────────────────┘          └──────────────────────────────────┘
```

:::info IPv4 vs IPv6
IPv4 ermöglicht etwa 4,3 Milliarden eindeutige Adressen, während IPv6 eine nahezu unbegrenzte Anzahl (340 Sextillionen, 36 Nullen) an Adressen bereitstellt. Der Umstieg auf IPv6 ist notwendig, da die IPv4-Adressen erschöpft sind.
:::

* **Private Adressbereiche**: Aufgrund der begrenzten Zahl von IPv4 Adressen gibt es Adressbereiche, die für private Netzwerke reserviert sind, um die Adressknappheit zu umgehen.
* **Nur lokal**: Diese Adressen sind nicht im Internet erreichbar, sondern nur innerhalb eines privaten Netzwerks.
* **NAT (Network Address Translation)**: Um diese privaten Adressen im Internet zu nutzen, wird NAT verwendet, um sie in öffentliche IP-Adressen umzuwandeln.
* **Router**: Dazu wird ein Router benötigt, der die Adressen übersetzt und die Pakete entsprechend weiterleitet.

### Domain Name System (DNS)

* **DNS** ist ein hierarchisches System zur Übersetzung von Domainnamen in IP-Adressen.
* **Benutzerfreundlich**: Nutzer geben in der Regel Domainnamen (z.B. www.example.com) ein, die dann in die entsprechende IP-Adresse aufgelöst werden.
* **Merkbare Namen**: Die Relevanz für Web Technologien liegt darin, dass DNS die Benutzerfreundlichkeit des Internets verbessert, indem es die Verwendung von leicht merkbaren Domainnamen anstelle von numerischen IP-Adressen ermöglicht.
* IPv4 Adressen sind in der Regel noch einigermaßen merkbar, aber IPv6 Adressen sind für Menschen kaum noch zu merken.
    * **IPv4 Format**: IPv4 Adressen werden numerisch in 4 Segmenten von 0 bis 254 unterteilt dargestellt
        * Beispiel: 127.0.0.1 (localhost, die eigene Maschine, auch Loopback-Adresse genannt)
    * **IPv6 Format**: IPv6 Adressen werden hexadezimal in 8 Segmenten von 0 bis FFFF unterteilt dargestellt
        * Beispiel: 2001:0db8:85a3:0000:0000:8a2e:0370:7334
* **DNS-Anfrage**: Um eine Domain in eine IP-Adresse aufzulösen, sendet der Client eine Anfrage an einen DNS-Server.
* **Datenbank-Lookup**: Der DNS-Server sucht die zugehörige IP-Adresse in seiner Datenbank und sendet sie zurück an den Client.
* **Hierarchische Weiterleitung**: Wenn der DNS-Server die IP-Adresse nicht kennt, kann er die Anfrage an andere DNS-Server weiterleiten, bis die Adresse gefunden wird.

```
  ┌──────────────┐
  │   Browser    │  1. Nutzer gibt ein: www.example.com
  │   (Client)   │
  └──────┬───────┘
         │
         │ 2. DNS-Anfrage: "Was ist die IP von www.example.com?"
         │
         ▼
  ┌──────────────────────┐
  │   Lokaler DNS-Server │  3. Suche in lokaler Datenbank...
  │   (z.B. vom Provider)│     └─▶ Nicht gefunden!
  └──────┬───────────────┘
         │
         │ 4. Weiterleitung an übergeordneten DNS-Server
         │
         ▼
  ┌──────────────────────┐
  │  Root DNS-Server     │  5. Verweist auf zuständigen Server
  │                      │     für .com-Domains
  └──────┬───────────────┘
         │
         │ 6. Anfrage an .com DNS-Server
         │
         ▼
  ┌──────────────────────┐
  │  .com DNS-Server     │  7. Verweist auf DNS-Server für
  │                      │     example.com
  └──────┬───────────────┘
         │
         │ 8. Anfrage an example.com DNS-Server
         │
         ▼
  ┌──────────────────────┐
  │ example.com          │  9. IP-Adresse gefunden!
  │ DNS-Server           │     93.184.216.34
  └──────┬───────────────┘
         │
         │ 10. Antwort: 93.184.216.34
         │
         ▼
  ┌──────────────┐
  │   Browser    │  11. Verbindung zu 93.184.216.34
  │              │      wird hergestellt
  └──────────────┘

```

### Transmission Control Protocol

* **TCP** ist **verbindungsorientiert**
* **Handshake**: Vor dem Datentransfer muss eine logische Verbindung zwischen Sender und Empfänger aufgebaut werden (Handshake).
* **Zuverlässige Übertragung**: Das Hauptziel ist die zuverlässige, geordnete und fehlerfreie Übertragung von Datenpaketen.
* **Ergänzt IP**: Es ergänzt das Internet Protocol (IP), das für die Adressierung und das Routing von Datenpaketen verantwortlich ist, aber selbst keine Garantie für die Zustellung oder Reihenfolge der Pakete bietet.
* **TCP-Pakete** enthalten Sequenznummern, um die Reihenfolge der Pakete zu verfolgen, und Prüfsummen, um die Integrität der Daten zu überprüfen.
* **TCP** ist wichtig für Web Technologien, da es die Grundlage für die meisten Internetprotokolle bildet, einschließlich HTTP und HTTPS.

```
    Sender                                  Empfänger
      │                                         │
      │  Paket 1 (Seq: 1, Daten: "Hallo")       │
      ├────────────────────────────────────────▶│
      │                                         │ ✓ Empfangen
      │ ◀───────────────────────────────────────┤
      │         ACK 1 (Bestätigung)             │
      │                                         │
      │  Paket 2 (Seq: 2, Daten: " Welt")       │
      ├─────────────────────✗ verloren          │
      │                                         │
      │  Paket 3 (Seq: 3, Daten: "!")           │
      ├────────────────────────────────────────▶│
      │                                         │ ✓ Empfangen
      │                                         │ ✗ Fehlt Paket 2!
      │ ◀───────────────────────────────────────┤
      │         ACK 1 (erneut angefordert)      │
      │                                         │
      │  Paket 2 (Seq: 2, erneut gesendet)      │
      ├────────────────────────────────────────▶│
      │                                         │ ✓ Komplett!
      │ ◀───────────────────────────────────────┤
      │         ACK 3 (Alles empfangen)         │
      │                                         │
      Ergebnis: "Hallo Welt!" (korrekte Reihenfolge)
```

### Hypertext Transfer Protocol (HTTP)

* **HTTP** ist das Protokoll, das für die Übertragung von Webseiten und Web Applikationen im Internet verwendet wird.
* Es ermöglicht den Austausch von Daten zwischen Webservern und Webbrowsern.
* **HTTP-Anfragen** werden von einem Client (z.B. Webbrowser) an einen Server gesendet, um Ressourcen wie HTML-Dokumente, Bilder oder Skripte anzufordern.
* **HTTP-Antworten** enthalten die angeforderten Ressourcen sowie Statuscodes, die den Erfolg oder Misserfolg der Anfrage anzeigen.
* **HTTP** ist ein zustandsloses Protokoll, was bedeutet, dass jede Anfrage unabhängig von vorherigen Anfragen behandelt wird.
* HTTP wird im Folgenden noch ausführlicher behandelt, da es das zentrale Protokoll für dieses Modul ist.

### Hypertext Transfer Protocol Secure (HTTPS)

* **HTTPS** ist die sichere Variante von HTTP, die eine verschlüsselte Kommunikation zwischen Client und Server ermöglicht.
* Es verwendet Transport Layer Security (TLS), um die Daten während der Übertragung vor Abhören und Manipulation zu schützen.
* **HTTPS** ist besonders wichtig für Web Applikationen, die sensible Daten übertragen, wie z.B. Online-Banking oder E-Commerce-Websites.
* Die Verwendung von HTTPS ist heute Standard und wird von den meisten modernen Webbrowsern und Webservern unterstützt und aktiv gefordert.
* TLS (Transport Layer Security) ist der Nachfolger von SSL (Secure Sockets Layer), SSL wird heute nicht mehr verwendet.
* Das Thema HTTPS und TLS wird im Kapitel "Web Application Security" tiefgehender behandelt.

## 2. HTTP im Detail

* HTTP funktioniert nach einem klaren **zustandslosen Prinzip**:
* **Request**: Der Client sendet eine Anfrage an den Server.
* **Response**: Der Server verarbeitet die Anfrage und sendet eine Antwort zurück.

* Eine HTTP-Anfrage besteht aus:
    * **Methode**: Gibt die Art der Anfrage an (z.B. GET, POST, PUT, DELETE).
    * **URL**: Die Adresse der angeforderten Ressource.
    * **HTTP-Version**: Gibt die verwendete Version des HTTP-Protokolls an (z.B. HTTP/1.1).
    * **Header**: Zusätzliche Informationen über die Anfrage (z.B. Content-Type, User-Agent).
    * **Cookies**: Kleine Datenpakete (Key-Value-Paare), die im Browser gespeichert werden können und im `Cookie` Header automatisch an den Server geschickt werden.
    * **Body**: Nur bei POST- oder PUT-Anfragen enthalten, enthält die zu sendenden Daten.

* Eine HTTP-Antwort besteht aus:
    * **HTTP-Version**: Gibt die verwendete Version des HTTP-Protokolls an (z.B. HTTP/1.1).
    * **Statuscode**: Ein dreistelliger Code, der den Erfolg oder Misserfolg der Anfrage anzeigt (z.B. 200 für Erfolg, 404 für nicht gefunden).
    * **Statusnachricht**: Eine kurze Beschreibung des Statuscodes.
    * **Header**: Zusätzliche Informationen über die Antwort (z.B. Content-Type, Content-Length).
    * **Body**: Enthält die angeforderten Daten (z.B. HTML-Dokument, JSON-Daten).

### HTTP Methoden

* Die wichtigsten HTTP-Methoden in der Praxis sind:
    * **GET**: Fordert eine Ressource an (z.B. eine Webseite).
        * **Daten abrufen**: Wird verwendet, um Daten vom Server abzurufen.
        * **Kein Body**: Keine Daten im Body der Anfrage, da es sich um eine reine Abfrage handelt.
        * Die Antwort enthält die angeforderten Daten im Response Body.
        * **Idempotent & Safe**: GET-Anfragen sollten idempotent sein, d.h. sie sollten keine Seiteneffekte haben und mehrfaches Ausführen sollte stets die gleiche Antwort liefern, solange die Resource sich nicht (durch andere Requests) verändert hat.
    * **POST**: Sendet Daten an den Server (z.B. Formulardaten).
        * **Daten erstellen**: Wird verwendet, um Daten zu erstellen oder zu aktualisieren.
        * **Mit Body**: Kann Daten im Body der Anfrage enthalten, die an den Server gesendet werden.
    * **PUT**: Aktualisiert eine Ressource auf dem Server.
        * **Vollständig ersetzen**: Wird verwendet, um eine Ressource vollständig zu ersetzen.
        * Kann Daten im Body der Anfrage enthalten, die die aktualisierte Ressource darstellen.
    * **PATCH**: Aktualisiert teilweise eine Ressource auf dem Server.
        * **Teilweise aktualisieren**: Wird verwendet, um eine Ressource teilweise zu aktualisieren.
        * Kann Daten im Body der Anfrage enthalten, die die Änderungen an der Ressource darstellen.
    * **DELETE**: Löscht eine Ressource auf dem Server.
        * **Ressource löschen**: Wird verwendet, um eine Ressource zu löschen.
        * **Kein Body**: Keine Daten im Body der Anfrage, da es sich um eine Löschoperation handelt.
  * Weitere HTTP-Methoden, die in der Praxis häufig beobachtet werden, sind:
    * **HEAD**: Fordert nur die Header einer Ressource an, ohne den Body.
        * Wird verwendet, um Metadaten über eine Ressource abzurufen.
        * Nützlich, um Informationen wie den Content-Type oder die Last-Modified-Zeit zu erhalten, ohne die gesamte Ressource herunterzuladen.
        * In der Praxis wird HEAD oft nicht manuell verwendet, sondern von Browsern oder Caches automatisch genutzt, um Informationen über Ressourcen zu erhalten.
    * **OPTIONS**: Fordert die unterstützten HTTP-Methoden und -Header für eine Ressource an.
        * Wird verwendet, um herauszufinden, welche HTTP-Methoden und -Header für eine bestimmte Ressource unterstützt werden.
        * Nützlich für CORS (Cross-Origin Resource Sharing) Anfragen, um zu überprüfen, ob eine Ressource von einer anderen Domain aus zugänglich ist.
        * Kann auch verwendet werden, um die Serverkonfiguration zu überprüfen.
        * In der Praxis wird OPTIONS oft von Browsern oder Caches automatisch genutzt, um Informationen über Ressourcen zu erhalten.

### HTTP Header

* **Metadaten**: Header sind Metadaten einer HTTP-Anfrage oder -Antwort.
* **Zusatzinformationen**: Sie begleiten den eigentlichen Inhalt (Body) und liefern Zusatzinformationen, z.B:
    * Wer sendet die Anfrage?
    * Welches Format hat der Inhalt?
    * Welche Caching Regeln gelten?
* **Key-Value-Paare**: Jeder Header ist ein Key-Value-Paar, das durch einen Doppelpunkt getrennt ist.
* Header sind case-insensitive (technisch), aber Konvention ist `My-Header-Name`
* **Reihenfolge egal**: Reihenfolge der Header ist nicht relevant.
* Klassische Request-Header:
    * `Host`: Ziel-Hostname (z.B. www.example.com)
    * `User-Agent`: Informationen über den Client (z.B. Browser, Betriebssystem)
    * `Accept`: Akzeptierte Antwortformate (z.B. `application/json`, `text/html`, siehe MIME-Typen)
    * `Authorization`: Authentifizierungsinformationen (z.B. Token, Basic Auth)
* Klassische Response-Header:
    * `Content-Type`: Format des Inhalts (z.B. `text/html`, `application/json`)
    * `Content-Length`: Länge des Inhalts in Bytes
    * `Set-Cookie`: Setzt Cookies für den Client (siehe Cookies)
    * `Cache-Control`: Caching-Regeln für die Antwort
    * `Location`: Bei Weiterleitungen (3xx) die neue URL, zu der der Client umgeleitet werden soll

### HTTP Cookies

* **Zustandsverwaltung**: Der Server kann den Client anweisen, sich bestimmte Informationen zu merken, die bei zukünftigen Anfragen automatisch mitgesendet werden sollen.
* **Set-Cookie Header**: Dies geschieht über den `Set-Cookie` Header in der HTTP-Antwort.
* Seltener kann auch der Client selbst Cookies setzen, die dann bei zukünftigen Anfragen an den Server gesendet werden.
* **Cookie Header**: Der Client speichert diese Informationen und sendet diese zukünftig im `Cookie` Header bei Anfragen an den "gleichen" Host.
* Häufige Einsatzzwecke:
    * **Session-Management** (Login-Status, Warenkorb, Nutzer-ID, Access-Tokens)
    * **Personalisierung** (Spracheinstellungen, Nutzerpräferenzen)
    * **Tracking** (Analyse von Nutzerverhalten, Werbung)
* Unterschiede zu Headern
    * **Automatisch**: Cookies werden automatisch an den Host geschickt. Dies geschieht, ohne dass diese explizit im Request definiert werden müssen.
    * **Persistent**: Cookies sind persistent, d.h. sie können über mehrere Sitzungen hinweg gespeichert werden.
    * **Attribute**: Cookies können zusätzliche Attribute haben, die ihr Verhalten steuern, z.B. `HttpOnly`, `Secure`, `SameSite`.
    * **Sicherheit**: Cookies können sicherer sein, da bspw. `HttpOnly` verhindert, dass JavaScript auf den Cookie zugreifen kann, was XSS-Angriffe erschwert.

:::danger Sicherheit bei Cookies
Cookies ohne `HttpOnly` und `Secure` Flags sind anfällig für XSS-Angriffe und Man-in-the-Middle-Attacken. Für Session-Cookies sollten **immer** `HttpOnly`, `Secure` und `SameSite=Strict` (oder `Lax`) gesetzt werden.
:::

* Technische Funktionsweise
    * **Setzen** eines Cookies:
        * **Serverseitig**: HTTP-Header `Set-Cookie: name=value; Path=/; HttpOnly; Secure; SameSite=Strict`
        * **Clientseitig**: JavaScript `document.cookie = "name=value; Path=/; Secure; SameSite=strict";`
        * Erklärung der Bestandteile:
            * `name=value`: Name und Wert des Cookies
            * `Path`: Der Pfad, für den das Cookie gültig ist (z.B. `/` für die gesamte Domain)
            * `HttpOnly`: Das Cookie ist nur über HTTP(S) zugänglich, nicht über JavaScript
            * `Secure`: Das Cookie wird nur über HTTPS-Verbindungen gesendet
            * `SameSite`: Einschränkung, wie das Cookie bei Cross-Site-Anfragen gesendet wird (z.B. `Strict`, `Lax`, `None`, siehe Tabelle unten)
    * **Senden** eines Cookies:
        * **Automatisch**: Bei jeder Anfrage an den gleichen Host sendet der Browser den Cookie im `Cookie` Header mit.
        * Beispiel: `Cookie: session=abc123; lang=de`
    * **Lebensdauer**:
        * **Session-Cookie** (ohne `Expires` oder `Max-Age`): Gültig bis zum Schließen des Browsers
        * **Persistent-Cookie** (mit Ablaufdatum): Gültig bis zum angegebenen Datum oder der angegebenen Zeitspanne
    * **Scope**:
        * **Domain**: Das Cookie ist für die angegebene Domain und alle Subdomains gültig.
        * **Path**: Das Cookie ist nur für den angegebenen Pfad und dessen Unterverzeichnisse gültig.

## 3. URI und URL

* **URI** (Uniform Resource Identifier)
    * **Oberbegriff**: Oberbegriff für alles, was eine Ressource im Internet eindeutig identifiziert.
    * Zwei Haupttypen:
        * **URL** (Uniform Resource Locator): Gibt den Ort der Ressource an und enthält Protokoll, Host, Pfad und optional Abfrageparameter (z.B. `https://www.example.com/path/to/resource?query=param`).
            * **Spezialfall**: URL ist also ein Spezialfall einer URI
            * Jede URL ist eine URI, aber nicht jede URI ist eine URL.
            * In der Praxis werden die Begriffe oft synonym verwendet, da URLs die am häufigsten verwendeten URIs sind.
        * **URN** (Uniform Resource Name): Identifiziert eine Ressource durch einen Namen, ohne ihren Ort anzugeben (z.B. `urn:isbn:0451450523` ISBN für Bücher).
            * Für dieses Modul irrelevant, daher nicht weiter behandelt.
* Aufbau einer absoluten URL
    * Ausführliches Beispiel: `scheme://user:password@host:port/path?query#fragment`
    * **scheme**: Protokoll, das verwendet wird (z.B. `http`, `https`, `ftp`).
    * **user:password**: _Optional_, für Authentifizierung, sieht man oft bei FTP- oder SSH-URLs, aber selten bei HTTP.
    * **host**: Domain oder IP-Adresse des Servers (z.B. `www.example.com`).
    * **port**: _Optional_, Portnummer, auf dem der Server lauscht (z.B. `:80` für HTTP, `:443` für HTTPS). Wenn nicht angegeben, wird der Standardport des Schemas verwendet.
    * **path**: _Optional_, Pfad zur Ressource auf dem Server (z.B. `/path/to/resource`). Streng genommen optional. Standardverhalten ist vom Protokoll abhängig.
    * **query**: _Optional_, Abfrageparameter, die zusätzliche Informationen über die Ressource liefern (z.B. `?query=param&another=value`).
    * **fragment**: _Optional_, Anker oder Abschnitt innerhalb der Ressource (z.B. `#section1`), wird oft in HTML-Dokumenten verwendet, um auf bestimmte Abschnitte zu verweisen.
* **Relative URI-Referenz**
    * umgangssprachlich oft auch als relative URL bezeichnet
    * **Kontextabhängig**: Eine relative URI-Referenz ist eine Adressangabe, die nicht alle Komponenten einer absoluten URL enthält, sondern vom Kontext (**Base-URL**) abhängig ist.
    * Die Base-URL wird in der Regel bestimmt durch:
        * die URL der aktuellen Seite, wenn die relative URI-Referenz in einem HTML-Dokument verwendet wird
        * die URL des Servers, wenn die relative URI-Referenz in einer API-Anfrage verwendet wird
        * Das `<base href="...">` Tag, das im `<head>` Bereich eines HTML-Dokuments verwendet werden kann, um die Base-URL für alle relativen URI-Referenzen im Dokument festzulegen.
    * Beispiel:
        * **Base-URL**: `https://www.example.com/`
        * **Relative URI-Referenz**: `/path/to/resource`
        * In diesem Fall würde die vollständige URL `https://www.example.com/path/to/resource` sein.
* **Origin / Basis-URL**
    * **Origin**: Die **Origin** ist die Kombination aus Protokoll, Host und Port einer URL.
    * **CORS**: Sie wird verwendet, um Sicherheitsrichtlinien wie CORS (Cross-Origin Resource Sharing) zu implementieren.
    * Beispiel: Für die URL `https://www.example.com:443/path/to/resource?query=param#section1` ist die Origin `https://www.example.com:443`.
    * **Sicherheitsmechanismus**: Die Origin ist wichtig für Sicherheitsmechanismen, die den Zugriff auf Ressourcen von verschiedenen Ursprüngen (Origins) steuern.
    * **Ohne Pfad**: Die Origin enthält keinen Pfad, keine Query-Parameter und keine Fragmente.
    * Wenn man nicht gerade im Kontext von relativen URI-Referenzen ist, spricht man manchmal auch von Basis-URLs, wenn man die Kombination aus Origin plus Pfad meint.
    * Diese Basis-URL enthält aber auch keine Query-Parameter und keine Fragmente

## 4. MIME Types / Internet Media Types und Content Negotiation

* **Strukturiert & kodiert**: Textbasierte Daten sind im Rahmen von digitaler Kommunikation in der Regel strukturiert und kodiert.
* **Format austauschen**: Um diese Daten korrekt zu interpretieren, ist es wichtig, dass die Kommunikationsteilnehmer (Client und Server) austauschen, in welchem Format die Daten übertragen werden.
* Dafür hat sich das Konzept der **MIME Types** (Multipurpose Internet Mail Extensions) etabliert.
* **Ursprung E-Mail**: Dieser kommt ursprünglich aus der E-Mail-Welt, wurde aber später von HTTP übernommen.
* Im Rahmen von HTTP spricht man auch von **Internet Media Types**.
* Der Aufbau eines MIME Types funktioniert wie folgt:
    * `type/subtype[; parameter]`
    * `type` stellt den Haupttypen dar. Dieser kann einer der Folgenden sein: `text`, `image`, `audio`, `video`, `application`, `multipart`, `message`
        * `text`: Textbasierte Daten
        * `image`: Bilddateien
        * `audio`: Audiodateien
        * `video`: Videodateien
        * `application`: Anwendungsdaten, die nicht in die anderen Kategorien passen (z.B. JSON, XML, PDF)
        * `multipart`: Mehrteilige Nachrichten, die mehrere Teile enthalten (z.B. Formulardaten)
        * `message`: Nachrichtenformate, die für die Übertragung von Nachrichten verwendet werden (z.B. E-Mails)
    * `subtype` spezifiziert den genauen Typ innerhalb des Haupttyps (z.B. `html`, `json`, `jpeg`, `png`)
        * Eine vollständige Liste würde sämtliche Rahmen sprengen.
        * Eine vollständige Liste lässt sich bei der [Internet Assigned Numbers Authority (IANA)](https://www.iana.org/assignments/media-types/media-types.xhtml) finden.
    * `parameter` ist optional und kann zusätzliche Informationen enthalten, z.B. `charset` für die Zeichencodierung bei Textdateien.
    * Beispiele:
        * text/plain; charset=UTF-8
        * image/jpeg
* **MIME Types im Rahmen von HTTP**
    * **Datentyp kennzeichnen**: MIME Types werden in HTTP-Anfragen und -Antworten verwendet, um den Typ der gesendeten oder empfangenen Daten zu kennzeichnen.
    * **Content-Type Header**: Um das Format der gesendeten Daten anzugeben, wird der MIME Type im `Content-Type` Header der HTTP-Anfrage oder -Antwort verwendet.
        * Beispiel: `Content-Type: application/json`
    * **Accept Header**: In HTTP-Anfragen kann außerdem im `Accept` Header angegeben werden, welche Formate der Client akzeptiert.
        * Beispiel: `Accept: application/json, application/xml`
        * Dies bedeutet, dass der Client JSON und XML akzeptiert, wobei JSON bevorzugt wird.
    * **Content Negotiation**: Das Aushandeln, mit welchem `Content-Type` die Daten in der Antwort übertragen werden, wird als **Content Negotiation** bezeichnet.
    * Content Negotiation ermöglicht es dem Client und Server, das am besten geeignete Format für die Kommunikation auszuhandeln.
    * Der Client nennt seine Präferenzen im `Accept` Header, und der Server wählt die passende Repräsentation aus.


## 5. REST

* **REST** (Representational State Transfer) ist ein Architekturstil für die Gestaltung von verteilten Systemen, insbesondere Web APIs.
* **Prinzipien**: Er beschreibt Prinzipien, wie man Ressourcen über das HTTP-Protokoll zugänglich macht und mit ihnen interagiert.
* **Kernidee**: Statt eigene Protokolle oder komplexe Standards zu erfinden, nutzt REST konsequent die vorhandenen Mechanismen von HTTP (Methoden, URIs, Statuscodes, Header, ...).
* Beispiel: Ein Dienst stellt eine Bibliothek dar.
    * **Ressourcen**: Ressourcen sind Bücher, Autoren, Kategorien, ...
    * **Eindeutige URI**: Jede Ressource hat eine eindeutige URI (z.B. `/books/123` für ein Buch mit der ID 123).
    * **HTTP-Methoden**: Der Client kann über HTTP-Methoden (GET, POST, PUT, DELETE) auf diese Ressourcen zugreifen und sie manipulieren.
        * Man kann das Buch ansehen (GET)
        * ein neues Hinzufügen (POST)
        * ein Bestehendes austauschen (PUT)
        * oder es aus dem Regal nehmen (DELETE)


### Motivation und Einordnung

* **Mehrere Komponenten**: Web-Applikationen bestehen oft aus mehreren Komponenten (Frontend und Backend, App und Server, ...).
* **Vertragliche Schnittstellen**: APIs sind die vertraglichen Schnittstellen, über die diese Komponenten miteinander kommunizieren.
* **Unabhängige Entwicklung**: Vorteil: System können unabhängig voneinander entwickelt werden, solange der Vertrag (API) eingehalten wird.
* **Früher SOAP**: Früher wurde oft SOAP (Simple Object Access Protocol) verwendet, ein XML-basiertes Protokoll, das entgegen des Versprechens im Namen komplex und schwergewichtig ist.
* SOAP wird heute noch in spezifischen Anwendungsfällen verwendet, z.B. in Unternehmensumgebungen oder bei Legacy-Systemen
* **Leichter & flexibler**: REST ist eine leichtere, flexiblere Alternative, die auf den bestehenden HTTP-Standards aufbaut.
* **De-facto Standard**: Heute ist REST der de-facto Standard für Web APIs.
* **Alternativen**: Neben REST und SOAP gibt es auch andere Architekturstile wie GraphQL, gRPC oder WebSockets, die in bestimmten Szenarien Vorteile bieten können.
* REST ist jedoch am weitesten verbreitet und wird in den meisten modernen Webanwendungen verwendet

### Grundprinzipien

#### Ressourcenorientierung
* **Alles ist Ressource**: Alles, was über die API zugänglich ist, wird als Ressource betrachtet.
* **Eindeutige URIs**: Ressourcen sind eindeutig über URIs identifizierbar.
    * Beispiel: `/books/123` für ein Buch mit der ID 123
* In objektorientierten Programmiersprachen entspricht eine Ressource oft einem Objekt, auf denen flexibel Methoden definier- und aufrufbar sind
* **Begrenzte Methoden**: In REST sind die Methoden auf die HTTP Methoden (`GET`, `POST`, ...) begrenzt.

##### Funktionsorientierte APIs (RPC):
```
POST /getUser?id=42
POST /createUser
POST /deleteUser?id=42
```

* Verben (wie `get`, `create`, `delete`) werden in der URL verwendet, um die Aktion zu beschreiben
* Jede Funktion ist eine eigene Schnittstelle für sich
* Dadurch wird die API unübersichtlich und schwerer verständlich, da jede Funktion eine eigene URL hat

##### Ressourcenorientierte APIs (REST):
```
GET /users/42
POST /users
DELETE /users/42
```

* Ressourcen werden über Nomen (wie `users`, `books`) identifiziert
* In diesem Beispiel gibt es nur eine Ressource `users`, die verschiedene Aktionen unterstützt
* Aktionen werden über die HTTP-Methoden (`GET`, `POST`, `DELETE`) definiert und nicht über den Pfad der URL
* Dadurch wird die API konsistenter und leichter verständlich
* Oft können REST-APIs "erraten" werden, weil die Methoden in der Regel den CRUD-Operationen (Create, Read, Update, Delete) entsprechen:
    * `GET` für Lesen (Read)
    * `POST` für Erstellen (Create)
    * `PUT` oder `PATCH` für Aktualisieren (Update)
    * `DELETE` für Löschen (Delete)

#### Einheitliche Schnittstelle

* **Gleichartige Interaktion**: Interaktion mit Ressourcen erfolgt immer gleichartig über HTTP Mechanismen.
* **HTTP-Methoden**: HTTP-Methoden bestimmen die Art der Interaktion.
* **Statuscodes**: Spezifizierte Statuscodes geben Rückmeldung über den Erfolg oder Misserfolg der Anfrage.
* **Konsequenz**: Konsistenz, einfache Lernbarkeit.

#### Zustandslosigkeit

* **Alle Infos enthalten**: Jede Anfrage vom Client an den Server muss alle Informationen enthalten, die der Server benötigt, um die Anfrage zu verstehen und zu verarbeiten.
* **Kein Zustand**: Der Server speichert keinen Zustand zwischen den Anfragen (außer die Ressourcen selbst).
* **Skalierbarkeit**: Vorteil: Skalierbarkeit, weil der Server beliebig ausgetauscht/lastverteilt werden kann.

#### Repräsentationen

* **Verschiedene Formate**: Ressourcen sind nicht gleich Datenbank-Tabellen oder Objekte - sie können in verschiedenen Repräsentationen vorliegen (z.B. JSON, XML, HTML).
* **Accept Header**: Der Client kann über den `Accept` Header angeben, welche Repräsentation er bevorzugt.
* **Content-Type Header**: Der Server kann über den `Content-Type` Header angeben, in welchem Format die Antwort gesendet wird.

#### Layered System (Schichtenarchitektur)

* **Transparent**: Der Client muss nicht wissen, ob er direkt mit dem Server spricht oder über Proxy, Cache, Loader Balancer, o.Ä.
* Durch die Zustandslosigkeit und Ressourcenorientierung funktionieren Load Balancing und Cache transparent

### Ressourcenmodellierung

* Was ist eine Ressource?
    * **Fachliches Abbild**: Eine Ressource ist ein Abbild eines fachlichen Objekts oder Konzepts
        * Beispiel: Ein Buch, ein Autor, eine Kategorie, ...
    * **Nicht gleich DB-Tabelle**: Eine Ressource ist nicht gleiche eine Datenbank Tabelle, aber oft gibt es eine Nähe
        * Beispiel: Eine Datenbank Tabelle `book` könnte die Ressource `books` repräsentieren
    * **Kanonische URI**: Jede Ressource ist eindeutig über eine primäre ("offizielle", kanonische) URI identifizierbar
        * Beispiel: `/books/123` für ein Buch mit der ID 123
        * **Alternative URIs**: Ein Buch könnte auch über andere URIs erreichbar sein, z.B. `/books?title=Mein%20Buch` oder `/books/author/42`, aber die primäre URI ist immer die kanonische URI
* **Nomen statt Verben**: Ressourcen werden über Nomen und nicht über Verben abgebildet
    * ❌ `/getUser`
    * ✅ `/users`
* **Sammlungen vs. Einzelressourcen**: Man unterscheidet zwischen "Sammlungen" und "Einzelressourcen"
    * **Plural**: Sammlungen werden immer im Plural benannt
        * Beispiel: `/books` für eine Sammlung von Büchern
    * **Identifier**: Einzelressourcen werden meistens über den Identifier der Ressource identifiziert
        * Beispiel: `/books/123` für das Buch mit der ID 123
* **Hierarchisch**: Ressourcen können hierarchisch strukturiert sein
    * **Sub-Ressourcen**: Dabei spricht man von "Sub-Ressourcen"
    * **Beziehungen**: Damit können bspw. Beziehungen zwischen Ressourcen abgebildet werden
    * Beispiel: `/library/42/books/4` für das Buch mit der ID 4 in der Bibliothek mit der ID 42
    * **Hypermedia**: Beziehungen können auch über Verlinkungen abgebildet werden (Hypermedia)
        * Beispiel: `/books/123` könnte einen Link zu `/authors/42` enthalten, um den Autor des Buches anzuzeigen
        * Am Rande erwähnt: Hypermedia und HATEOAS (Hypermedia as the Engine of Application State) sind Konzepte, die in RESTful APIs verwendet werden können, um Ressourcen dynamisch zu verknüpfen und den Client durch die API zu navigieren.
* **Klar & konsistent**: Ressourcen werden klar, konsistent und lesbar benannt
    * Beispiel: `/books`, `/authors`, `/categories`
    * **Keine Abkürzungen**: Vermeidung von Abkürzungen, kryptischen IDs oder technischen Begriffen
* **Versionierung**: Ressourcen können versioniert werden, um Änderungen an der API (dem Vertrag) zu ermöglichen
    * Beispiel: `api/v1/books`, `api/v2/books`
    * **URL-Versionierung**: Versionierung kann auch über Header oder Query-Parameter erfolgen, aber die URL-Versionierung ist am gebräuchlichsten

### HTTP Methoden im Kontext von REST

* REST APIs definieren keine eigenen Methoden für Ressourcen, sondern nutzen die HTTP-Methoden
* Die HTTP-Methoden werden verwendet, um die Aktionen auf den Ressourcen zu definieren
* HTTP Methoden können `safe` oder `idempotent` sein
    * `safe` bedeutet, dass die Methode keine Seiteneffekte hat, d.h. sie ändert den Zustand der Ressource nicht
    * `idempotent` bedeutet, dass die Methode bei mehrfacher Ausführung das gleiche Ergebnis liefert, d.h. sie hat keine zusätzlichen Seiteneffekte

* **GET**
    * **Lesen**: Liest eine Ressource oder eine Sammlung von Ressourcen
    * Das Aufrufen einer Ressource mit GET hat keine Seiteneffekte (`safe`)
    * Beispiel: `GET /books` für eine Liste aller Bücher oder `GET /books/123` für ein bestimmtes Buch
* **POST**
    * **Erstellen**: Erstellt eine neue Ressource
    * **Sammlung erweitern**: Wird oft verwendet, um eine Sammlung zu erweitern
    * **Generelles Verb**: POST kann auch als "generelles" Verb verwendet werden, um eine Aktion auf einer Ressource auszuführen, die nicht durch GET, PUT, PATCH oder DELETE abgedeckt ist
    * **REST-ig bleiben**: Wichtig: "REST-ig" bleiben; Es sollte stets in Ressourcen gedacht werden und nicht ins RPC-Denken abrutschen
    * Man spricht dabei auch von "operation as resource"
    * Beispiel:
        * ❌ POST /orders/123/cancel

        * ✅ Ressourcenorientiert modellieren:
            * POST /orders/123/cancellations → erzeugt eine Cancellation-Ressource
* **PUT**
    * **Vollständig**: Aktualisiert eine Ressource vollständig (idempotent)
    * **Gesamte Ressource ersetzen**: Ersetzt die gesamte Ressource durch die im Body der Anfrage gesendeten Daten
    * Beispiel: `PUT /books/123` mit den neuen Daten für das Buch mit der ID 123
* **PATCH**
    * **Teilweise**: Aktualisiert eine Ressource teilweise
    * **Bestimmte Felder**: Erlaubt es, nur bestimmte Felder der Ressource zu aktualisieren
    * PATCH wird meistens idempotent implementiert
        * kann als "setze Feld X auf Wert Y" verstanden werden: idempotent
        * kann aber auch als "addiere Wert Y zu Feld X" verstanden werden: nicht idempotent
    * Beispiel: `PATCH /books/123` mit den geänderten Feldern für das Buch mit der ID 123
* **DELETE**
    * **Löschen**: Löscht eine Ressource
    * Wird meistens idempotent implementiert, d.h. das mehrfache Löschen derselben Ressource hat keine weiteren Auswirkungen
    * Beispiel: `DELETE /books/123` für das Löschen des Buches mit der ID 123

* Vollständiges Beispiel:
    * **GET** `/books` → Liste aller Bücher
    * **GET** `/books/123` → Details zum Buch mit der ID 123
    * **POST** `/books` → Neues Buch erstellen
    * **PUT** `/books/123` → Buch mit der ID 123 vollständig aktualisieren
    * **PATCH** `/books/123` → Teilweise Aktualisierung des Buches mit der ID 123
    * **DELETE** `/books/123` → Buch mit der ID 123 löschen

### Status Codes im Kontext von REST

* REST nutzt die HTTP-Statuscodes, um den Erfolg oder Misserfolg einer Anfrage anzuzeigen
* Wie bereits gelernt, sind die HTTP-Statuscodes klar definiert und standardisiert
* Deswegen ergeben sich kaum Ergänzungen zu dem Thema
* Beispiele im Kontext von REST:
    * `POST /users` → `201 Created` (Ressource erfolgreich erstellt)
    * `GET /users/42` → `200 OK` (Ressource erfolgreich abgerufen)
    * `PUT /users/42` → `200 OK` (Ressource erfolgreich aktualisiert) oder `204 No Content` (Ressource erfolgreich aktualisiert, aber keine Daten zurückgegeben)
    * `DELETE /users/42` → `204 No Content` (Ressource erfolgreich gelöscht)
    * `GET /users/999` → `404 Not Found` (Ressource nicht gefunden)
    *  `POST /users` mit ungültigen Daten → `400 Bad Request` (Anfrage war ungültig)

### Best Practices (für Einsteiger)

* **APIs = Verträge**: APIs sind Verträge zwischen Client und Server
    * **Leicht zu brechen**: Es ist sehr einfach bei der Entwicklung dieser API den Vertrag zu brechen
    * Beispiele:
        * ❌ Das Verschieben einer Ressource von `/users` nach `/people` ist ein Breaking Change
        * ❌ Das Entfernen eines Endpunktes wie `/users/42` ist ein Breaking Change
        * ❌ Das Ändern des Rückgabeformats von JSON zu XML ist ein Breaking Change
        * ✅ Das Hinzufügen eines neuen Endpunktes wie `/users/42/friends` ist kein Breaking Change
        * ✅ Das Hinzufügen eines neuen verpflichtenden Feldes in der Antwort von `/users/42` ist kein Breaking Change
        * ❌ Das Hinzufügen eines neuen verpflichtendes Feldes in der Anfrage von `/users` ist ein Breaking Change
        * ✅ Das Hinzufügen eines neuen optionalen Feldes in der Anfrage von `/users` ist kein Breaking Change
    * **Breaking Changes vermeiden**: Breaking Changes sollten vermieden werden, da sie bestehende Clients brechen können und somit zu Laufzeitfehlern führen
    * **Dokumentieren & kommunizieren**: Breaking Changes sollten immer gut dokumentiert und kommuniziert werden. Außerdem sollte den Clients genügend Zeit gegeben werden, um sich auf die Änderungen vorzubereiten.
    * **Von Anfang an stabil**: Aufgrund dieses Aufwandes ist es empfehlenswert, APIs so zu gestalten, dass sie von Anfang an stabil sind und nicht ständig geändert werden müssen.
    * **Leicht erweiterbar**: Außerdem sollten APIs so gestaltet werden, dass sie leicht erweiterbar sind, ohne dass bestehende Clients betroffen sind.
    * **Versionierung**: Zu guter Letzt sollte immer eine Versionierung der API in Betracht gezogen werden, um Breaking Changes zu vermeiden und den Clients die Möglichkeit zu geben, auf eine stabile Version der API zuzugreifen.
* Dokumentation
    * **Dokumentation = A & O**: Eine gute Dokumentation ist das A und O für eine API
    * **Vollständig**: Sie sollte alle Endpunkte, Parameter, Rückgabewerte und mögliche Fehlercodes beschreiben
    * **OpenAPI**: Es gibt verschiedene Tools und Standards, um APIs zu dokumentieren, z.B. OpenAPI (früher Swagger) oder RAML
    * In der Praxis wird oft OpenAPI verwendet, da es weit verbreitet ist und viele Tools unterstützt
        * Beispiel: https://api.mittwald.de/v2/openapi.json
    * **Maschinenlesbar**: OpenAPI ermöglicht es, die API in einem maschinenlesbaren Format zu beschreiben, das dann automatisch in eine lesbare Dokumentation umgewandelt werden kann
        * Beispiel: https://api.mittwald.de/v2/docs
        * Oder: https://developer.mittwald.de/docs/v2/reference
* Zusätzliche Informationen bei Fehlern liefern
    * **Detaillierte Fehlerinfos**: Bei Fehlern sollte die API so viel wie möglich Informationen liefern, um dem Client zu helfen, das Problem zu verstehen und zu beheben
    * Beispiel: Bei einem `400 Bad Request` sollte die API nicht nur den Statuscode zurückgeben, sondern auch eine Fehlermeldung und ggf. Details zu den fehlerhaften Feldern
    * Dies kann in der Antwort im Body als JSON-Objekt erfolgen, das Informationen über den Fehler enthält
    * **Konsistent modellieren**: Errors sollten ebenso wie die erfolgreichen Antworten konsistent modelliert werden
    * Beispiel:
        ```json
        {
          "status": "error",
          "message": "Ungültige Anfrage",
          "errors": [
            {
              "field": "email",
              "type": "invalid_format",
              "message": "E-Mail-Adresse ist ungültig"
            },
            {
              "field": "password",
              "type": "required",
              "message": "Passwort muss mindestens 8 Zeichen lang sein"
            }
          ]
        }
        ```
